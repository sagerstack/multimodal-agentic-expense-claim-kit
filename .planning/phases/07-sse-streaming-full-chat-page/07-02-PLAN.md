  <name>Task 1: Jinja2 partial templates for SSE HTML fragments</name>
  <files>
    templates/partials/message_bubble.html
    templates/partials/summary_panel.html
  </files>
  <action>
    Create `templates/partials/` directory and two fragment templates. These are NOT full pages — they are HTML snippets rendered server-side and delivered as SSE `data` payloads for HTMX `sse-swap` insertion.

    **No thinking_panel.html needed.** The thinking panel is assembled on the frontend from raw text SSE events (step-name, step-content, thinking-done). The runGraph generator yields plain text for these events, and HTMX `sse-swap` inserts them directly into the DOM elements in chat.html. No server-side template rendering is involved for the thinking panel.

    **message_bubble.html** — A single chat message bubble. Accepts template variables:
    - `content` (str): The message text (markdown-safe, but plain text for now)
    - `isAi` (bool): True for AI messages, False for user messages
    - `timestamp` (str, optional): Display timestamp

    Structure:
    - AI messages: left-aligned with bot icon (`smart_toy`), `bg-surface-container-low` background, `rounded-2xl rounded-tl-none`
    - User messages: right-aligned, `bg-primary/10` background, `rounded-2xl rounded-tr-none`
    - Both wrapped in `<div class="flex gap-4 max-w-2xl ...">` matching the greeting bubble style in chat.html
    - Content rendered in `<p class="text-on-surface text-sm leading-relaxed">` with `whitespace-pre-wrap` to preserve newlines
    - Timestamp as `<span class="text-[10px] text-outline px-1">{{ timestamp }}</span>`
    - For AI messages with confidence scores: support optional `confidenceScores` dict — if present, render as small inline badges `<span class="text-[10px] bg-secondary/10 text-secondary px-2 py-0.5 rounded-full">merchant: 99%</span>` after the content text (CHAT-10)
    - For policy violations: support optional `violations` list — if present, render each as `<p class="text-[10px] text-tertiary italic mt-1">{{ violation }}</p>` (CHAT-11)

    **summary_panel.html** — Content for the submission summary bento grid. Accepts:
    - `totalAmount` (str): e.g., "$4.50" or "SGD 4.50"
    - `itemCount` (int): Number of items
    - `topCategory` (str): e.g., "meals"
    - `warningCount` (int): Number of policy violation flags
    - `progressPct` (int): 0-100
    - `batchItems` (list[dict], optional): Each has `merchant`, `amount`, `category`

    Structure: Matches the existing summary panel layout in chat.html — replaces the inner content of the bento grid and batch details. Uses the same Tailwind classes already in chat.html. The `sessionTotal`, `sessionItemCount`, `sessionProgress` elements' content is now rendered server-side instead of static placeholders.
  </action>

    Each file is valid HTML (no unclosed tags) — verify with a quick visual scan
  </verify>
  <done>

No thinking_panel.html — thinking panel content is plain text delivered via SSE and assembled on the frontend.
  </done>

<task type="auto">
  <name>Task 2: Enhance runGraph with template rendering, summary extraction, and interrupt wiring</name>
  <files>
    src/agentic_claims/web/sseHelpers.py
  </files>
  <action>
    Enhance the `runGraph` generator in sseHelpers.py (created in Plan 01) with three capabilities:

    **1. Message bubble rendering (MIGR-04, CHAT-01):**
    After the astream_events loop completes and thinking-done is yielded, render the final AI response into an HTML fragment:
    ```python
    finalText = _stripThinkingTags(_stripToolCallJson(tokenBuffer)) if tokenBuffer.strip() else await _getFallbackMessage(graph, config)
    ```
    Extract the inline fallback logic from Plan 01's runGraph into a dedicated `_getFallbackMessage` helper. Plan 01 already includes this fallback logic inline within runGraph (state extraction if buffer empty, same as app.py lines 353-376). This task refactors it into a named helper for clarity and reuse. If finalText exists, render via `templates.get_template("partials/message_bubble.html").render(content=finalText, isAi=True)` and yield as MESSAGE event.

    **2. Summary panel extraction (CHAT-06, CHAT-07, CHAT-08):**
    Add a new helper `_extractSummaryData(thinkingEntries: list) -> dict | None` that scans thinkingEntries for tool outputs:
    - From `extractReceiptFields` output: extract `merchant`, `totalAmount`, `currency`, `category` fields
    - From `searchPolicies` output: count policy violations as warnings
    - From `convertCurrency` output: extract converted amount
    - From `submitClaim` output: mark item as submitted

    Build a summary dict: `{"totalAmount": ..., "itemCount": ..., "topCategory": ..., "warningCount": ..., "progressPct": ..., "batchItems": [...]}`. After THINKING_DONE is yielded, if summary data exists, render `templates.get_template("partials/summary_panel.html").render(**summaryData)` and yield as SUMMARY_UPDATE event.

    **3. Interrupt detection (MIGR-05, CHAT-04):**
    After the astream_events loop, check for interrupt:
    ```python
    finalState = await graph.aget_state(config={"configurable": {"thread_id": threadId}})
    if finalState.next:
        for task in finalState.tasks:
            if hasattr(task, "interrupts") and task.interrupts:
                payload = task.interrupts[0].value
                question = payload.get("question", str(payload))
                request.session["awaiting_clarification"] = True
                yield ServerSentEvent(data=question, event=SseEvent.INTERRUPT)
    ```
    When interrupt is detected, do NOT yield the MESSAGE event — the clarification question IS the response.

    **4. Image message formatting:**
    In `buildGraphInput` (add this helper if not in Plan 01), when `graphInput["hasImage"]` is True, format the HumanMessage content exactly as app.py does:
    - With user text: `f'User says: "{message}"\n\nI\'ve also uploaded a receipt image for claim {claimId}. Please process it using extractReceiptFields.'`
    - Without user text: `f"I've uploaded a receipt image for claim {claimId}. Please process it using extractReceiptFields. No expense description was provided."`

    **5. Confidence score and violation pass-through (CHAT-10, CHAT-11):**
    In the MESSAGE rendering, if `extractReceiptFields` output exists in thinkingEntries, extract per-field confidence scores (if present in the VLM output) and pass as `confidenceScores` dict to the template. If `searchPolicies` found violations, extract the cited clauses and pass as `violations` list to the template.
