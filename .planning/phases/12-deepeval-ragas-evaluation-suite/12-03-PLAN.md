  <name>Task 1: Implement deterministic BaseMetric subclasses</name>
  <files>eval/src/metrics/deterministic.py</files>
  <action>
Create `eval/src/metrics/deterministic.py` with custom `BaseMetric` subclasses for the 8 deterministic benchmarks. Each subclass must implement `measure()`, `a_measure()`, `is_successful()`, and `__name__` property. Must set `self.score`, `self.success`, `self.reason` in `measure()`.

All metrics receive data via `testCase.additional_metadata` (a dict containing the captured + expected output).

**1. DocumentTypeMetric** (ER-001, ER-002, ER-003):
- Checks if the agent's classification matches expected document type
- `additional_metadata["agentDecision"]` compared against `additional_metadata["expectedDecision"]`
- Normalize: strip, lowercase, check for keyword containment (e.g., "receipt" in output for ER-001, "not a receipt" or "unsupported" or "needs review" for ER-002/003)
- Score: 1.0 (match) or 0.0 (no match)

**2. ReceiptCompletenessMetric** (ER-004):
- Checks if required fields (merchant, date, amount) are present in extracted fields
- `additional_metadata["extractedFields"]` must have non-null, non-empty values for "merchant", "date", "total" (or "amount" or "totalAmount")
- Score: count of present fields / 3

**3. FieldExtractionMetric** (ER-005, ER-006, ER-010):
- Compares extracted fields against expected fields
- For each field in `additional_metadata["expectedFields"]`:
  - String fields: case-insensitive, strip whitespace, check containment (merchant name might be abbreviated)
  - Numeric fields (total, subtotal, tax): compare with tolerance of 0.01
  - Date fields: normalize to YYYY-MM-DD before comparison
- Score: matching fields / total expected fields

**4. AmountReconciliationMetric** (ER-015):
- Compares extracted total against claimed amount from companion metadata
- `additional_metadata["extractedFields"]["total"]` vs `additional_metadata["companionMetadata"]["claimedAmount"]`
- Score: 1.0 if absolute difference <= 0.01, else 0.0
- Reason includes both values

Each metric class:
- Constructor takes `threshold: float = 1.0` (default: must be perfect)
- CamelCase class and method names
- `a_measure` delegates to `measure` (deterministic, no async needed)

Also create a factory function `getDeterministicMetric(benchmarkId: str) -> BaseMetric` that returns the appropriate metric instance for the benchmark ID.

**eval/src/metrics/semantic.py** -- GEval instances for 10 semantic benchmarks:

Create a `getSemanticMetric(benchmarkId: str, judgeModel: LiteLLMModel) -> GEval` factory function.

Each GEval uses `evaluation_steps` (NOT `criteria` -- they are mutually exclusive per research). Use `LLMTestCaseParams.ACTUAL_OUTPUT` and `LLMTestCaseParams.EXPECTED_OUTPUT` as evaluation params. Set `threshold=0.7`.

Define evaluation steps per benchmark:

- **ER-007, ER-008** (Category Assignment):
  ```python
  evaluation_steps=[
      "Check if the assigned expense category matches the expected category",
      "Verify the category assignment is supported by evidence from the receipt content (merchant name, items purchased)",
      "Assess whether the category label uses the correct taxonomy term",
  ]
  ```

- **ER-009** (Reimbursable vs Non):
  ```python
  evaluation_steps=[
      "Check if the agent correctly identified this document as a non-standard receipt (invoice/AP document)",
      "Verify the agent routed it for review rather than processing as a standard employee expense",
      "Check if the reasoning mentions workflow routing or document type distinction",
  ]
  ```

- **ER-011** (Itemization Readiness):
  ```python
  evaluation_steps=[
      "Check if the agent correctly identifies the receipt as having multiple line items",
      "Verify the agent detects the multi-item structure (more than one product/service listed)",
      "Assess whether individual line items are recognized with quantities and prices",
  ]
  ```

- **ER-012** (Receipt to Expense Entry Matching):
  ```python
  evaluation_steps=[
      "Check if the agent correctly matches or rejects the receipt against the expense entry",
      "Verify the matching decision uses date, amount, and vendor agreement",
      "Assess whether partial matches are handled appropriately (e.g., date matches but amount differs)",
  ]
  ```

- **ER-013** (Duplicate Detection):
  ```python
  evaluation_steps=[
      "Check if the agent correctly identifies this as a duplicate or near-duplicate submission",
      "Verify the agent references the original claim or prior receipt",
      "Confirm the claim was flagged, rejected, or escalated rather than approved",
  ]
  ```

- **ER-014** (Date Window Compliance):
  ```python
  evaluation_steps=[
      "Check if the agent evaluates the receipt date against the submission policy window",
      "Verify the decision (in-policy or late submission) is correct for the given date",
      "Assess whether the agent cites the relevant date policy rule",
  ]
  ```

- **ER-016** (Approval Routing):
  ```python
  evaluation_steps=[
      "Check if the agent correctly routes the claim based on its total amount",
      "Verify the routing decision (standard vs senior approval) matches the threshold rules",
      "Assess whether the agent explains the threshold that triggered the routing decision",

  ```

- **ER-017** (Out of Policy Spend):


**eval/src/metrics/safety.py** -- HallucinationMetric for ER-018 ONLY:

Create `getHallucinationMetric(judgeModel: LiteLLMModel) -> HallucinationMetric`:
```python
from deepeval.metrics import HallucinationMetric

def getHallucinationMetric(judgeModel):
    return HallucinationMetric(
        threshold=0.1,  # score > 0.1 = hallucination detected = FAIL
        model=judgeModel,
        include_reason=True,
    )
```

CRITICAL: HallucinationMetric uses `context` field on LLMTestCase, NOT `retrieval_context`. The `context` for ER-018 must be sourced from `benchmark["groundTruthFacts"]` (the known facts about what receipt 18.pdf actually contains), NOT from `retrievedPolicyChunks`. Policy chunks describe expense rules -- they are NOT the source of truth for what fields exist on a receipt. Document this in a comment.

**eval/src/metrics/retrieval.py** -- Native retrieval metrics for ER-009, ER-014, ER-017 ONLY:

**IMPORTANT**: ER-018 does NOT get retrieval metrics. ER-018 is a hallucination/safety benchmark where the agent should abstain. If the agent correctly abstains, there is no meaningful actual_output to evaluate retrieval quality against. Retrieval metrics are only applicable to benchmarks where the agent actively reasons using retrieved policy context.

Create `getRetrievalMetrics(judgeModel: LiteLLMModel) -> list[BaseMetric]`:
```python
from deepeval.metrics import (
    ContextualPrecisionMetric,
    ContextualRecallMetric,
    FaithfulnessMetric,
)

def getRetrievalMetrics(judgeModel):
    return [
        ContextualPrecisionMetric(threshold=0.7, model=judgeModel),
        ContextualRecallMetric(threshold=0.7, model=judgeModel),
        FaithfulnessMetric(threshold=0.7, model=judgeModel),
    ]
```

Do NOT import from `deepeval.metrics.ragas` -- those wrappers are hardcoded to OpenAI model strings only.

These metrics use `retrieval_context` field on LLMTestCase (the policy chunks from Qdrant enrichment).
