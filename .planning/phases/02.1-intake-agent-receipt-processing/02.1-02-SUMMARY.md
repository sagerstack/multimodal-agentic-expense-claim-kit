# Phase 02.1 Plan 02: Intake Agent Tools and Dual Currency Schema Summary

**One-liner:** Four LangGraph tools (policy search, currency conversion, claim submission, HITL clarification) with dual currency database migration enabling multi-currency expense claims.

## What Was Built

Built the complete Intake Agent toolbox (4 new tools + 1 from Plan 01 = 5 total) and database schema for dual currency tracking.

**Policy Search Tool (searchPolicies):**
- RAG MCP server integration via mcpCallTool
- Returns policy chunks with text, file, category, section, and relevance score
- Configurable result limit (default: 5 chunks)
- Graceful error handling (returns error dict, no exceptions)

**Currency Conversion Tool (convertCurrency):**
- Currency MCP server integration via mcpCallTool
- Returns originalAmount, convertedAmount, rate, and date
- Live exchange rates for SGD conversion
- Supports any currency pair (fromCurrency → toCurrency)

**Claim Submission Tool (submitClaim):**
- DB MCP server integration with dual-call pattern
- First call: insertClaim (returns claim_id)
- Second call: insertReceipt with claim_id as foreign key
- Returns both claim and receipt records
- Error handling prevents orphaned receipts (FK integrity)

**Human Clarification Tool (askHuman):**
- LangGraph interrupt() primitive for HITL flow
- Synchronous (blocks until user responds)
- Passes question and context data to user
- Returns user action and corrected data
- Enables selective clarification for low-confidence fields

**Dual Currency Database Schema:**
- Alembic migration 002 adds 6 columns (3 per table)
- Claims: originalCurrency, originalAmount, convertedAmountSgd
- Receipts: originalCurrency, originalAmount, convertedAmountSgd
- All columns nullable (existing records preserved)
- Downgrade removes all 6 columns cleanly

**All 5 Intake Agent Tools:**
1. extractReceiptFields (from Plan 01) - VLM extraction
2. searchPolicies - RAG policy lookup
3. convertCurrency - Currency conversion
4. submitClaim - Database persistence
5. askHuman - Human-in-the-loop clarification

## Decisions Made

**1. submitClaim Dual-Call Pattern**
- **Context:** Need to insert claim and receipt with FK link
- **Decision:** Call insertClaim first, then insertReceipt with returned claim_id
- **Rationale:** Ensures FK integrity, prevents orphaned receipts, atomic from tool perspective
- **Impact:** No separate transaction management needed, MCP server handles DB atomicity

**2. Nullable Dual Currency Columns**
- **Context:** Adding new columns to existing tables with data
- **Decision:** All 6 dual currency columns nullable
- **Rationale:** Existing claims don't have currency conversion data, migration must not break them
- **Impact:** Application logic must handle None values for old claims

**3. askHuman Synchronous Interrupt**
- **Context:** Need to pause agent for human input
- **Decision:** Use LangGraph interrupt() (synchronous, blocking)
- **Rationale:** Standard LangGraph HITL primitive, well-documented, integrates with checkpointing
- **Impact:** Agent execution pauses until user responds via Chainlit UI

**4. All Tools Use MCP Client Utility**
- **Context:** 3 tools need to call different MCP servers
- **Decision:** All use shared mcpCallTool utility from Plan 01
- **Rationale:** Consistent error handling, single point of maintenance for MCP protocol
- **Impact:** Uniform error response format across all tools

## Knowledge for Future Phases

**Tool Usage Patterns:**
- searchPolicies: Agent calls when user asks about policy rules (meal limits, documentation requirements)
- convertCurrency: Agent calls when receipt currency ≠ SGD (company standard)
- submitClaim: Agent calls after all validation passes and user confirms
- askHuman: Agent calls when VLM confidence < threshold (from Plan 01 config: 0.7)

**Dual Currency Storage Pattern:**
- If receipt.currency == "SGD": store in totalAmount, leave dual currency fields NULL
- If receipt.currency != "SGD": store original in originalCurrency/originalAmount, converted in convertedAmountSgd
- Application logic must handle both patterns (NULL dual currency = original was SGD)

**Claim Submission Flow:**
1. Agent extracts receipt (extractReceiptFields)
2. If currency != SGD, agent converts (convertCurrency)
3. If low confidence, agent clarifies (askHuman)
4. Agent checks policy compliance (searchPolicies)
5. Agent submits claim and receipt (submitClaim)

**MCP Server Dependencies:**
- RAG MCP: Must be running for policy search
- Currency MCP: Must be running for non-SGD receipts
- DB MCP: Must be running for claim submission
- All URLs configured in Settings (from Plan 01)

## Architecture Insights

**Tool Wrapping Pattern:**
All MCP tools follow same structure:
1. Import @tool decorator and mcpCallTool utility
2. Define async function with typed parameters
3. Get settings inside function body (not at module level)
4. Call mcpCallTool with server URL, tool name, arguments
5. Return result directly (mcpCallTool handles errors)

**FK Integrity via Dual-Call:**
submitClaim ensures FK integrity without DB transaction management:
- First call establishes claim record with auto-generated claim_id
- Second call uses claim_id as foreign key
- If first call fails, second never executes
- If second call fails, orphaned claim exists but is incomplete (no receipt)
- Future: Consider DB MCP transaction support for true atomicity

**HITL Integration Points:**
askHuman enables selective clarification at multiple points:
1. After VLM extraction (low confidence fields)
2. Before policy check (ambiguous categories)
3. Before submission (user confirmation)
