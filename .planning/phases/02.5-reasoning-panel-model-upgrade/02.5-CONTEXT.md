# Phase 2.5 Context: Reasoning Panel + Model Upgrade

## Execution Instructions

### Browser-Based Verification (MANDATORY)

Each plan in this phase MUST include browser-based verification as a completion gate. Implementation is NOT complete until the requirement is verified in the running application.

**Protocol:**

1. After implementing code changes for a plan, rebuild and restart the app (`./scripts/startup.sh --reset` or `docker compose up --build`)
2. Use the Chrome browser automation tools (`mcp__claude-in-chrome__*`) to:
   - Navigate to `http://localhost:8000`
   - Upload a test receipt image
   - Interact with the claim flow
   - Verify each requirement visually and functionally
3. For each success criterion covered by the plan, verify:
   - The feature works as specified
   - The UI renders correctly (light mode, dark mode where applicable)
   - No errors in the browser console or Docker logs
4. Document verification results in the plan's SUMMARY.md
5. If verification fails, fix the issue and re-verify before marking the plan complete

**Verification must cover all 9 objectives:**

| Objective | What to Verify in Browser |
|-----------|--------------------------|
| 1. Type A+B reasoning | Expand thinking panel — LLM reasoning text visible before tool calls, QwQ reasoning tokens visible when present |
| 2. Progressive streaming | Thinking panel appears immediately when agent starts working, content streams in real-time |
| 3. QwQ-32B model | Agent responds successfully using QwQ-32B (check Docker logs for model name) |
| 4. Light/dark mode CSS | Toggle theme — thinking panel readable in both modes with proper contrast |
| 5. Schema-driven mapping | Thinking panel shows getClaimSchema call as first tool, agent maps fields dynamically |

**Only after ALL applicable objectives for a plan are verified in-browser will the plan execution be considered complete.**

### UAT

User Acceptance Testing (`/gsd:verify-work`) will be conducted separately AFTER all plans in this phase are executed and self-verified. The browser verification during execution is the builder's own quality gate — UAT is the user's independent validation.

## Decisions Made (from UAT analysis)

### Model Selection
- **Primary LLM**: `qwen/qwq-32b` via OpenRouter ($0.15/1M input, $0.58/1M output)
- **Reasoning**: Competitive with DeepSeek R1 at ~4-5x lower cost, 131K context window
- **Fallback strategy**: Graceful degradation — fallback model shows Type A reasoning only if it lacks reasoning tokens

### Reasoning Architecture
- **Type A**: Agent intermediate text (`AIMessage.content` before tool calls) — available for all models
- **Type B**: Model reasoning tokens (QwQ's `reasoning_content` field) — QwQ-specific
- **Display**: Interleaved chronologically — reasoning block, then tool call, then reasoning block, etc.
- **Streaming**: Progressive — thinking panel appears immediately with "Thinking...", content streams in as events arrive

### System Prompt Design (v2)
- **Acknowledgment**: Agent starts each turn with brief acknowledgment (immediate user feedback)
- **Schema-first**: Call `getClaimSchema` first to discover required fields dynamically
- **Field mapping**: Map extracted receipt fields to schema, infer where logical, ask user for gaps
- **Currency**: Call `convertCurrency` for EACH monetary value — never calculate manually
- **Justification**: Captured in Phase 2 summary table when policy violation exists
- **Remarks**: User description from upload captured and shown in summary
- **Full prompt**: See system prompt v2 design in conversation history

### New Tool Required
- **`getClaimSchema`**: Returns DB schema for claims and receipts tables (required/optional fields, types)
- **Implementation**: New MCP tool on DB server querying `information_schema.columns`, or LangChain tool wrapping `executeQuery`

### Architecture Changes
- **app.py streaming**: Shift from buffer-first to progressive rendering
- **app.py event handler**: Capture `tokenBuffer` before clearing (Type A), parse reasoning tokens (Type B)
- **Thinking panel HTML**: New structure with reasoning blocks interleaved between tool entries
- **Temperature**: Reduce from 0.7 to lower value for reasoning model (0.3 recommended)
