# Phase 6.3: User Authentication with Dual Roles

## Goal

Add user authentication to the application with two roles — **user** (claimant who submits expenses) and **reviewer** (approver who reviews escalated claims). Unauthenticated visitors see a login page matching the Stitch "Login - Expense AI" design. After login, users are routed to the Chat Page and reviewers to the Dashboard. Role-based access controls restrict page visibility by role.

## Depends on

Phase 6.2 (Chat UI complete, employee ID extraction working)

## Scope

1. **Login page** — Jinja2 template matching Stitch design (`docs/ux/stitch-login-expense-ai.html`)
2. **User model** — Database table for users with username, hashed password, role (user/reviewer), employee_id
3. **Authentication middleware** — Session-based auth, login/logout endpoints, redirect unauthenticated to /login
4. **Role-based routing** — Users see Chat page, Reviewers see Dashboard/Audit/Review pages
5. **Sidebar: Dashboard item** — Only visible to reviewer role; hidden for user role
6. **Seed data** — Pre-configured test users for both roles
7. **Employee ID from auth** — Replace manual employee ID extraction with authenticated user's employee_id

## UX Reference

- Login page design: `docs/ux/stitch-login-expense-ai.html` (Stitch "Login - Expense AI")
- Dark theme, glass panel, neon glow focus, gradient CTA button
- Fields: Username, Password, Remember device checkbox
