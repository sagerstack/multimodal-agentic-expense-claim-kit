<task type="auto">
  <name>Task 1: Review data router with HTML page handler, claim detail, and decision endpoints</name>
  <files>
    src/agentic_claims/web/routers/review.py
    src/agentic_claims/web/routers/pages.py
    src/agentic_claims/web/main.py
    tests/test_review.py
  </files>
  <action>
    1. Create `src/agentic_claims/web/routers/review.py`:

       - `GET /review/{claimId}` -> **HTML page handler** (moved from pages.py). This serves the review page with full server-side data:
         - Import `getCurrentUser` from auth.py, check `role == "reviewer"` (redirect to `/` if not)
         - Query claim with receipts, parse intake_findings for flag reason, query employee history for AI insight -- all server-side
         - Pass to template context: `claim`, `receipt`, `flagReason`, `aiInsight`, `userRole`, `displayName`, `employeeId`, `username`, `activePage="review"`
         - Render `templates/review.html`

       - `GET /api/review/{claimId}` -> returns JSON with full claim detail:
         ```
         {
           claim: {id, claimNumber, employeeId, status, totalAmount, currency, createdAt},
           receipt: {merchant, date, totalAmount, currency, imagePath, lineItems, originalCurrency, originalAmount, convertedAmountSgd},
           flagReason: {explanation: str, confidence: float} | null,
           aiInsight: {submissionCount: int, approvalRate: float, employeeName: str} | null
         }
         ```
         - `flagReason`: Parse from claims table intake_findings JSONB. Look for violations/flags. If no intake_findings or no flags, return null.
         - `aiInsight`: Query count of all claims by this employee_id, count of approved claims, calculate rate. Join with users table for display_name. If no prior claims, return null.

       - `POST /api/review/{claimId}/decision` -> accepts form data:
         - `action`: "approve" or "reject"
         - `rejectionReason`: string (required if action=reject, one of: "Duplicate submission", "Incomplete receipt details", "Policy violation")
         - `reviewerNotes`: string (optional free text)
         Update claim status in DB:
         - approve -> status = "approved", set approval_date = now()
         - reject -> status = "rejected"
         Create audit_log entry:
         - action: "claim_approved" or "claim_rejected"
         - actor: reviewer's display_name (from session)
         - new_value: JSON with {action, rejectionReason?, reviewerNotes?}
         Return redirect to `/dashboard` (302).

       - `GET /api/review/{claimId}/receipt-image` -> serve receipt image file.
         Query receipt for the claim, get image_path. The image_path may be a relative path under static/ or an absolute path. Serve as FileResponse or redirect to the static URL.

    2. **Remove the `/review/{claimId}` route from `src/agentic_claims/web/routers/pages.py`**. The review.py router now owns this route entirely. Leave all other routes in pages.py untouched.

    3. Include the review router in main.py (`app.include_router(reviewRouter)`).

    4. Create `tests/test_review.py`:
       - `testClaimDetailEndpointReturnsFullData`: Mock DB, verify JSON shape with claim, receipt, flagReason, aiInsight
       - `testApproveClaimUpdatesStatusAndCreatesAuditLog`: Mock DB, verify status change + audit log creation
       - `testRejectClaimRequiresReason`: Verify 422 or error if rejectionReason missing on reject action
       - `testRejectClaimWithReasonSucceeds`: Mock DB, verify status + audit log with reason
       - `testReceiptImageEndpoint`: Mock receipt with image_path, verify response
  </action>
  <verify>
    `poetry run pytest tests/test_review.py -v` -- all tests pass.
    With app running: `curl http://localhost:8000/api/review/1` returns claim detail JSON.
    Login as reviewer, navigate to `/review/1` -- page loads with full claim data (not empty template).
  </verify>
  <done>Review API endpoints return claim details with flag reason and AI insight, and process approve/reject decisions with audit logging. Review HTML page handler moved from pages.py to review.py with full server-side data loading.</done>

<task type="auto">
  <name>Task 2: Claim Review Jinja2 template matching Stitch design</name>
  <files>

  <action>
    1. Rewrite `templates/review.html` extending base.html. Match Stitch design at `docs/ux/stitch-claim-review-escalation.html`:

       **Layout:** Two-column split view:
       - Left column (~50%): Receipt image with zoom controls
       - Right column (~50%): Extracted fields, flag reason, AI insight, action form

    2. **Left column - Receipt Image:**
       - Header: "Original Receipt" with claim number
       - Receipt image displayed in a container with overflow hidden
       - Alpine.js zoom controls: `x-data="{ scale: 1 }"` 
         - Zoom In button (`+`): `@click="scale = Math.min(scale + 0.25, 3)"`
         - Zoom Out button (`-`): `@click="scale = Math.max(scale - 0.25, 0.5)"`
         - Reset button: `@click="scale = 1"`
         - Image: `:style="'transform: scale(' + scale + ')'"`
       - If no receipt image: show placeholder with "No receipt image available"

    3. **Right column - Extracted Entity Data:**
       - Section header: "Extracted Entity Data"
       - Read-only field cards in a grid (2 columns):
         - Merchant Name: icon `store`, value from receipt.merchant
         - Transaction Date: icon `calendar_today`, value from receipt.date
         - Amount: icon `payments`, value formatted with currency (show original + converted if applicable)
         - Category: icon `category`, value from line_items or "General"
       - Each card: dark glass background, icon, label, value. NOT editable.

    4. **Flag Reason card:**
       - Header: "Flagged" or "AI Assessment" with warning icon
       - Body: explanation text from flagReason.explanation
       - Confidence bar: visual bar showing flagReason.confidence percentage
       - Fallback: "No assessment data available" if flagReason is null

    5. **AI Insight card:**
       - Header: "AI Insight" with smart_toy icon
       - Body: "Employee has {submissionCount} previous submissions with {approvalRate}% approval rate"
       - Fallback: "No prior submission history available" if aiInsight is null

    6. **Action form:**
       - Reviewer Notes: textarea with placeholder "Add review notes..."
       - Rejection Reason: 3 radio buttons (hidden by default, shown when Reject path chosen):
         - Duplicate submission
         - Incomplete receipt details
         - Policy violation
       - Two buttons:
         - **Approve** (green): `hx-post="/api/review/{claimId}/decision"` with `action=approve`
         - **Reject** (red): disabled until a rejection reason radio is selected. Alpine.js: `x-data="{ rejectionReason: '' }"`, `:disabled="!rejectionReason"`
       - Both buttons include reviewer notes in the form data
       - On success: HTMX redirect to `/dashboard` (use `HX-Redirect` response header from server)

    7. The review page handler in review.py (Task 1) already passes all claim data server-side to the template context. Use `{{ claim.claimNumber }}`, `{{ receipt.merchant }}`, `{{ flagReason.explanation }}` etc. directly in Jinja2.

    Color scheme: Neon Nocturne dark theme throughout. Status badges same colors as dashboard.
