<objective>
Decouple post-submission agents from the SSE stream (BUG-026), add LLM timeout handling (BUG-025), and fix the stale Decision Pathway state (BUG-024). After the intake agent submits a claim, the chat immediately returns the success response. Compliance, fraud, and advisor agents run as a background asyncio task. The background task also writes the `ai_reviewed` status after compliance+fraud complete. If the LLM hangs during intake processing, the user sees an error toast after a configurable timeout.

Purpose: The current architecture blocks the SSE stream for 60-90s while compliance/fraud/advisor run. This makes the user experience feel broken. Decoupling means the user gets instant feedback after submission. The BUG-024 fix prevents stale checkpoint state from marking Decision Pathway steps as COMPLETED prematurely.
Output: Modified SSE pipeline that terminates early after submitClaim, background task runner for post-submission agents with ai_reviewed status write, BUG-024 stale pathway fix, and LLM timeout configuration.
</objective>

    **New background task function in sseHelpers.py (or a new file `src/agentic_claims/web/backgroundTasks.py`):**
    Create `runPostSubmissionAgents(graph, threadId: str, claimId: str)`:
    ```python
    async def runPostSubmissionAgents(graph, threadId: str, claimId: str):
        """Run compliance, fraud, and advisor agents in the background.
        
        Timing assumption: checkpoint was written after intake node completion
        (claimSubmitted=True). We verify this before proceeding.
        """
        config = {"configurable": {"thread_id": threadId}}
        try:
            # Checkpoint guard: verify the graph state has claimSubmitted=True
            # before attempting to resume. This confirms the checkpoint was
            # written after the intake node completed.
            currentState = await graph.aget_state(config)
            if not currentState.values.get("claimSubmitted"):
                logger.error(
                    "Checkpoint guard failed for claim %s: claimSubmitted is not True. "
                    "The checkpoint may not have been written after intake completion.",
                    claimId
                )
                return
            
            # Resume graph from checkpoint. evaluatorGate sees claimSubmitted=True
            # and routes to postSubmission -> compliance || fraud -> advisor.
            # After compliance+fraud complete and BEFORE advisor runs,
            # write ai_reviewed status to the DB.
            await graph.ainvoke(None, config=config)
        except Exception as e:
            logger.error("Background post-submission failed for claim %s: %s", claimId, e, exc_info=True)
    ```
    
    IMPORTANT: The graph execution model matters here. When we break out of `astream_events`, the graph run is interrupted. The checkpointer has saved state up to the intake node completion (with `claimSubmitted=True`). We need to re-invoke the graph so it picks up from the checkpoint and routes through evaluatorGate -> postSubmission -> compliance || fraud -> advisor.
    
    To re-invoke: call `graph.ainvoke(None, config=config)` -- passing None as input tells LangGraph to resume from the last checkpoint. The evaluatorGate sees `claimSubmitted=True` and routes to the post-submission pipeline.

    **ai_reviewed status write:**

This is cleaner because it's part of the graph topology.
    
    2. **In the background task:** After `graph.ainvoke` completes the compliance/fraud nodes, manually call updateClaimStatus. This is harder because `ainvoke` runs the full remaining graph (compliance+fraud+advisor) in one shot.

This node calls the DB MCP server's `updateClaimStatus` tool to set `status = "ai_reviewed"`. The advisor node then immediately overwrites with its final decision (ai_approved/ai_rejected/escalated).
