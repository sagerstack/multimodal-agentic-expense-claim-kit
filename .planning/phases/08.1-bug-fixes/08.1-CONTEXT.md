<domain>

## Domain

Phase 8.1 is a stabilization + UX polish phase. It fixes 3 open bugs from Phase 8 QA (Decision Pathway, LLM timeout, post-submission blocking), restructures the claim status lifecycle to an 8-state model, rebuilds the Claims page "My Claims" table, reorganizes the Claim Review page card layout, and decouples post-submission agents from the SSE stream.

The application is a multi-agent expense claims system with 4 LangGraph agents (intake, compliance, fraud, advisor). The web UI uses FastAPI + Jinja2 templates + HTMX + Alpine.js with SSE streaming for the chat interface.

</domain>

<decisions>

## Decisions

### 1. Claim Review Card Layout

**Decision:** Cards move from the sidebar to below the receipt details section, ordered by agent pipeline sequence.

- **Card order (top to bottom):**
  - Row 1: `Intake Agent Findings` | `Conversational Audit` (2-col)
  - Row 2: `Compliance Check` | `Fraud Analysis` (2-col)
  - Row 3: `AI Insight` (full width)
- **Receipt image + extracted fields** stay at the top of the page (current position), above all cards
- **Responsive:** 2-column grid on desktop, stacks to single column on mobile

### 2. Intake Agent Findings Card

**Decision:** Renamed from "Flag Reason". Shows two key metrics from `intake_findings.confidenceScores`:

- **Average confidence score** across all extracted fields (single number, color-coded green/yellow/red)
- **Lowest confidence field** highlighted with attribute name and value (e.g., "Category: Meals — 72%")
- Data source: `claims.intake_findings` JSONB column → `confidenceScores` key
- Note: Not all claims have `confidenceScores` populated (e.g., CLAIM-020 is missing it). The planner must ensure the intake agent always stores this. Card should gracefully handle missing data.

### 3. Conversational Audit Card

**Decision:** New card showing the intake agent's conversation with the claimant. Displays in **timeline format** with icon-prefixed entries.

**Captures (all four types):**
1. **Policy violations** — e.g., "Amount $52.30 exceeds SGD 40 taxi cap" (⚠ icon)
2. **User justifications** — the user's response to the violation (indented under the violation)
3. **Soft cap breaches** — warning-level observations (ℹ icon)
4. **Field corrections** — when the user corrected an extracted field (✏ icon)

**Display format:**
```
⚠ Policy: Amount $52.30 exceeds SGD 40 taxi cap (transport.md)
  └ User: Shared ride with 3 colleagues, split equally

ℹ Soft cap: Meal $38 approaching SGD 40 daily limit
  └ User: Client lunch meeting
```

**Data source:** `claims.intake_findings` JSONB column — already stores `policyViolation`, `justification`, `remarks`. No new column or migration needed. The planner should ensure the intake agent stores all 4 observation types consistently in `intake_findings`.

### 4. Pending Intelligence Cards (Background Agents)

**Decision:** When compliance/fraud/advisor haven't run yet, their cards show a **"Pending Review" status badge** with no content and a "Refresh page to check status" message.

- No auto-polling, no HTMX polling, no WebSocket
- User manually refreshes the page to see updated results
- Cards populate with full content once the background agent has written its findings to the DB

### 5. My Claims Table

**Decision:** Replace the current Claims page submission table with a "My Claims" section.

- **Header:** "My Claims" above the table
- **Filtering:** Only show claims belonging to the logged-in user (filter by `employee_id`)
- **Ordering:** Most recent first (descending by `created_at`)
- **Layout:** Table moves to the left, "Current Report" summary card is removed
- **Columns (7):** Claim #, Date, Merchant, Category, Original Amount, SGD Amount, Status
- **Row click behavior:**
  - Draft rows: not clickable (display only, greyed out or subtle styling)
  - Submitted claims: click navigates to **Audit Log** page (`/audit?claim_id={id}`)

### 6. Draft Claim in Table

**Decision:** Create a DB row with status `draft` as soon as the user sends their **first message** in the chat session.

- Row appears immediately in the My Claims table with partial data (`--` for missing fields like amount, merchant)
- As the intake agent processes (extraction, conversion), the draft row's fields get updated
- On submission, status transitions from `draft` → `pending`
- If the user abandons the chat session, the draft row remains in the DB (stale drafts are acceptable)

### 7. Claim Status Lifecycle

**Decision:** Replace the current flat statuses with an 8-state lifecycle:

```
draft → pending → ai_reviewed → ai_approved / ai_rejected / escalated → manually_approved / manually_rejected
```

| Status | Meaning | Set By |
|--------|---------|--------|
| `draft` | Claim being worked on in chat | Chat router (first message) |
| `pending` | Submitted to DB, awaiting AI pipeline | submitClaim tool |
| `ai_reviewed` | Compliance + fraud agents completed | Last of compliance/fraud agents |
| `ai_approved` | Advisor auto-approved (clean claim) | Advisor agent |
| `ai_rejected` | Advisor rejected (policy violations) | Advisor agent |
| `escalated` | Advisor escalated for human review | Advisor agent |
| `manually_approved` | Reviewer approved escalated claim | Reviewer action |
| `manually_rejected` | Reviewer rejected escalated claim | Reviewer action |

Impacts ~65 locations: advisor node, review router, dashboard KPIs, all templates, all tests.

### 8. Post-Submission Background Processing

**Decision:** Decouple post-submission agents from the SSE stream entirely.

- **Chat behavior:** After `submitClaim` persists the claim, the SSE stream immediately fires the "done" event with a success message: "Claim CLM-XXX submitted successfully!" + link to Audit Log page
- **Background execution:** Compliance, fraud, and advisor agents run as `asyncio.create_task` — completely detached from the user's chat session
- **No live updates:** The user does not see compliance/fraud/advisor progress in the chat
- **Claim Review page:** Shows "Pending Review" badges on intelligence cards until background agents complete. User refreshes manually to see results.

### 9. Bug Fixes (BUG-024, BUG-025, BUG-026)

**BUG-024 (Decision Pathway premature COMPLETED):**
- Root cause: `sseHelpers.py` lines 548-576 reconstructs pathway state from `graph.aget_state()` using stale checkpointed data
- Fix: Only mark steps as COMPLETED based on tool calls observed in the **current** SSE stream, not from prior checkpoint state

**BUG-025 (Chat stuck at Analyzing, no error toast):**
- Root cause: OpenRouter LLM call hangs indefinitely — no timeout configured
- Fix: Add timeout to OpenRouter client calls. Emit SSE error event if no `astream_events` output within N seconds (configurable via env var). Error surfaces as red toast in UI.

**BUG-026 (Post-submission agents block SSE stream):**
- Resolved by Decision #8 (background processing). The SSE stream no longer waits for the full graph pipeline.

## Implementation Specifics

### Data Source for Cards
- **Intake Agent Findings:** `claims.intake_findings` → `confidenceScores` (dict of field→float)
- **Conversational Audit:** `claims.intake_findings` → `policyViolation`, `justification`, `remarks` + new keys for soft cap breaches and field corrections
- **Compliance Check:** `claims.compliance_findings` (JSONB, already exists)
- **Fraud Analysis:** `claims.fraud_findings` (JSONB, already exists)
- **AI Insight:** `claims.advisor_decision` (varchar) + audit_log entries

### Draft Claim Creation
- Triggered in the chat router when first user message arrives
- Needs: `employee_id` from session (logged-in user), `status='draft'`, `total_amount=0`, minimal required fields
- The `claim_number` auto-generates via DB sequence (`claim_number_seq`)
- The `claimId` in the LangGraph state should reference this DB row from the start

### Status Column Width
- Current `status` column is `VARCHAR(20)` — `manually_approved` is 17 chars, fits. `manually_rejected` is 17 chars, fits.

### Existing intake_findings Structure
From CLAIM-019:
```json
{
  "remarks": "No expense description provided",
  "category": "Transportation",
  "conversion": { "date": "...", "rate": 1.2871, ... },
  "justification": "this was a limo ride prebooked by the hotel. pls proceed",
  "policyViolation": "Exceeds SGD 40.00 per-trip cap for taxi/ride-hail (Section 2.2)",
  "confidenceScores": { "date": 0.98, "currency": 0.99, "merchant": 0.95, "totalAmount": 0.99 }
}
```
Not all claims populate `confidenceScores` — the intake agent must be made to always include it.
