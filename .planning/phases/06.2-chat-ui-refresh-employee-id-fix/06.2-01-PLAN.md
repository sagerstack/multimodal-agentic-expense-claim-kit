**Architecture:** Use Python `contextvars.ContextVar` to inject the employee ID into `submitClaim.py` at the tool execution level, completely bypassing the LLM. The LLM never sees, propagates, or influences the employee ID value. This eliminates BUG-015's root cause (LLM ignoring/hallucinating employee IDs).

**1. Create context variable module (`src/agentic_claims/web/employeeIdContext.py`):**

```python
"""Request-scoped employee ID context variable.

Set by the chat router before graph invocation.
Read by submitClaim tool at MCP call time.
Uses Python contextvars for async-safe request isolation.
"""
from contextvars import ContextVar

employeeIdVar: ContextVar[str | None] = ContextVar("employeeIdVar", default=None)
```

**2. Chat router (`src/agentic_claims/web/routers/chat.py`):**

In `postMessage()`, after receiving the message text:
- Call `extractEmployeeId(message)` on every user message
- If it returns a non-None value, store it in `request.session["employee_id"]`
- BEFORE invoking the graph, set the context variable: `employeeIdVar.set(request.session.get("employee_id"))`

Import `extractEmployeeId` from `agentic_claims.web.employeeIdExtractor`.
Import `employeeIdVar` from `agentic_claims.web.employeeIdContext`.

The context variable is set once per request and is automatically scoped to the current async task — no cleanup needed, no cross-request leakage.

**3. submitClaim tool (`src/agentic_claims/agents/intake/tools/submitClaim.py`):**

At the top of the `submitClaim` function body, BEFORE building MCP arguments:
- Import and read the context variable: `from agentic_claims.web.employeeIdContext import employeeIdVar`
- Read: `serverEmployeeId = employeeIdVar.get(None)`
- If `serverEmployeeId` is not None, override `claimData["claimantId"]` with it:
  ```python
  if serverEmployeeId:
      logger.info("Server-side employee ID override", extra={"serverEmployeeId": serverEmployeeId, "llmProvidedId": claimData.get("claimantId")})
      claimData["claimantId"] = serverEmployeeId
  ```
- This happens BEFORE the CLAIM_FIELD_MAP mapping, so the override flows through to `mergedArgs["employeeId"]` naturally
- The LLM-provided `claimantId` is completely replaced — logged for debugging but never used

**4. System prompt (`agentSystemPrompt_v2.py`):**

- In Phase 1 step 8, change `"e.g., EMP-001"` to just `"your employee ID"` — remove the specific example that biases the model
- The agent still asks users for their employee ID (for UX clarity), but the system prompt should NOT instruct the agent to pass it to submitClaim — the server handles that now
