<objective>
Close UAT Gap 2 (policy-exception justification loop) by fixing the two root causes diagnosed in `13-DEBUG-policy-exception-loop.md`:

1. **F1 (H1 primary) — Clear `clarificationPending` when the pending question is answered.** The flag is currently set True on unsupported-currency `convertCurrency` returns (`postToolFlagSetter.py:149`) and never cleared. Once stale-True, the `postModelHook` drift predicate (`hasContent ∧ ¬hasToolCalls ∧ clarificationPending`) misclassifies every subsequent legitimate plain-text reply as drift, triggering a soft-rewrite → second drift → `validatorEscalate=True` → `humanEscalationNode` → claim status "escalated". This is ALSO Fix C from `13-DEBUG-display-regression.md` (cross-referenced — SAME fix, consolidated here to avoid duplication).

2. **F2 (H2 secondary) — Add Phase-2-after-justification instruction to v5 prompt.** `agentSystemPrompt_v5.py:165-169` (Section 4 Phase 2 step 5) tells the LLM to call askHuman for a justification and "do not advance until the user responds" — but there is NO step 6 telling the LLM what to do AFTER the user responds. Even with F1 alone, the LLM might stall or loop on the justification turn. F2 closes the prompt gap.

Out of scope for this plan (deferred as `TODO(post-13)` hardening):
- **F3** (first-class `policyExceptionJustified` ClaimState field + drift-predicate tightening): nice-to-have defence-in-depth; only needed if F1+F2 prove insufficient. Per debug doc guidance ("F3 and F4 as hardening once F1+F2 stabilize").
- **F4** (distinguishing validator-drift vs policy-exception-timeout in humanEscalation classifier).
- **Cross-claim stale-flag reset** (askHumanCount + all loop-bound flags reset on new claim). Mentioned in debug doc "Related behaviours / ripple risks". File separate debug note.

Purpose: Unblocks Phase 13 ship. Gap 2 prevents valid VND-with-policy-violation submissions from completing and lands them in ESCALATED instead.
