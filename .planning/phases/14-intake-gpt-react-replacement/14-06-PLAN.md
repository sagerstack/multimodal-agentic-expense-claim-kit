**State-machine design:**

```
field_confirmation (buttons: Yes/No)
   │
   ├─ Yes click → applyToolResultsNode sets currentStep="field_confirmation_answered"
   │              (Plan 14-03 Gate 1 calls searchPolicies)
   │
   └─ No click  → _classifyInterruptReply returns "side_question" for field_confirmation + negative token
                → applyToolResultsNode keeps pendingInterrupt set (side_question guard from 14-02)
                → BUT: this plan adds a SPECIAL branch — when pending.kind=="field_confirmation"
                       AND outcome=="side_question" AND the text is a negative token specifically,
                       the runtime re-classifies as "correction_requested" and emits a NEW
                       requestHumanInput(kind="field_correction") asking "What looks incorrect?"

   field_correction (free text, NO buttons)
   │
   └─ free-text reply → applyToolResultsNode parses the correction, updates slots,
                         sets currentStep="correction_received"
                      → reasonNode bypass (new, this plan) detects currentStep=="correction_received",
                         emits a NEW requestHumanInput(kind="field_confirmation") with the updated
                         extraction table in contextMessage — buttons re-render.

submit_confirmation (buttons: Yes/No)
   │
   ├─ Yes click → submitClaim (existing flow unchanged)
   │
   └─ No click  → _classifyInterruptReply returns "cancel_claim"
                → applyToolResultsNode clears pendingInterrupt, sets workflow.status="cancelled",
                   currentStep="submission_declined" (existing value)
                → reasonNode bypass (new, this plan) detects currentStep=="submission_declined"
                   AND workflow.status in {"cancelled","completed"} → emits plain AIMessage
                   "Claim cancelled. When you're ready to try again, upload a new receipt."
```

**Key design note:** The "No" button sends the literal text `"no"`. Plan 14-02's classifier returns:
- `field_confirmation` + "no" → `side_question` (Plan 14-02 line: "User indicated the extracted details are not correct.")
- `submit_confirmation` + "no" → `cancel_claim`

For field_confirmation, this plan narrows the "side_question" outcome: if the responseText matches a negative token EXACTLY (not an interrogative side question), treat it as a correction-requested signal distinct from a free-form side question. Update `_classifyInterruptReply` to return a new outcome `"correction_requested"` for this specific case.

```python
def _buildFieldConfirmationAfterCorrectionAiMessage(intakeState: IntakeGptState) -> AIMessage | None:
    # Reuse the existing _buildFieldConfirmationAiMessage but append correction context.
    base = _buildFieldConfirmationAiMessage(intakeState)
    if base is None or not base.tool_calls:
        return None
    correctionText = (intakeState.get("slots") or {}).get("correctionText") or ""
    if correctionText:
        # Append correction text to contextMessage so the user sees the pending edit.
        toolCall = dict(base.tool_calls[0])
        args = dict(toolCall.get("args") or {})
        originalContext = str(args.get("contextMessage") or "")
        args["contextMessage"] = (
            f"{originalContext}\n\nYou reported: \"{correctionText}\"\n"
            f"Does the updated extraction above look correct now?"
        )
        toolCall["args"] = args
        return AIMessage(content="", tool_calls=[toolCall])
    return base
```
