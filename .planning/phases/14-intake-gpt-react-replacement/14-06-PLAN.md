**State-machine design:**

```
field_confirmation (buttons: Yes/No)
   │
   ├─ Yes click → applyToolResultsNode sets currentStep="field_confirmation_answered"
   │              (Plan 14-03 Gate 1 calls searchPolicies)
   │
   └─ No click  → _classifyInterruptReply returns "correction_requested" for
                  field_confirmation + negative token (NEW outcome added in this plan)
                → applyToolResultsNode clears pendingInterrupt, sets
                  currentStep="field_correction_requested", status="active"
                → reasonNode bypass (new, this plan) detects the currentStep and
                  synthesizes requestHumanInput(kind="field_correction") asking
                  "What looks incorrect?"

   field_correction (free text, NO buttons)
   │
   └─ free-text reply → applyToolResultsNode parses the correction, updates slots,
                         sets currentStep="correction_received"
                      → reasonNode bypass (new, this plan) detects currentStep=="correction_received",
                         emits a NEW requestHumanInput(kind="field_confirmation") with the updated
                         extraction table in contextMessage — buttons re-render.

submit_confirmation (buttons: Yes/No)
   │
   ├─ Yes click → submitClaim (existing flow unchanged)
   │
   └─ No click  → _classifyInterruptReply returns "cancel_claim"
                → applyToolResultsNode clears pendingInterrupt, sets
                   workflow.status="cancelled", currentStep="submission_declined",
                   AND sets intakeState.sessionReset = True
                → reasonNode bypass (new, this plan) emits plain AIMessage:
                   "Claim cancelled. Upload a new receipt to start a new claim."
                → chat.py SSE loop detects intakeGpt.sessionReset and rotates the
                   session (new claimId, new threadId, clears imageStore) mirroring
                   the existing claimSubmitted reset path.
```

**Key design note — classifier outcome for field_confirmation No:**

Plan 14-02's classifier currently returns `side_question` for field_confirmation + negative token. This plan NARROWS that to a new outcome `"correction_requested"`, which is distinct from a free-form interrogative side question. Update `_classifyInterruptReply` accordingly:

- `field_confirmation` + "no"/"nope"/"not really"/"wrong" → `"correction_requested"` (handled by correction loop)
- `field_confirmation` + free-form interrogative ("what does X mean?") → `"side_question"` (handled by Plan 14-02 side-question guard)
- `submit_confirmation` + "no" → `"cancel_claim"` (unchanged from 14-02)

This narrowing changes the behaviour assumed by Plan 14-02's `must_haves` (which has been updated to scope the side-question preservation claim accordingly — see 14-02 must_haves risk note).

**Change 9 — `chat.py` session-reset handler**

Locate the existing `claimSubmitted` reset logic in `src/agentic_claims/web/routers/chat.py` (search for `claimSubmitted` in the SSE loop / `postMessage` path). That logic rotates the session by generating a new `claimId`, a new `thread_id`, and clearing the `imageStore` entry for the old claim.

Mirror the same logic for `sessionReset`. Use the **`priorState` in the next POST preamble** pattern — identical to how `claimSubmitted` is detected. The SSE loop does NOT have access to `finalState`; the detection must happen in `postMessage` when the NEXT user action arrives.

```python
# In postMessage preamble, AFTER the existing claimSubmitted rotation block:
priorState = await graph.aget_state({"configurable": {"thread_id": threadId}})
priorIntakeGpt = (priorState.values.get("intakeGpt") or {}) if priorState else {}

if priorIntakeGpt.get("sessionReset") is True:
    logEvent(
        logger,
        "chat.session_reset",
        logCategory="chat",
        claimId=claimId,
        threadId=threadId,
        message="submit_confirmation cancelled — rotating session via sessionReset flag",
    )
    imageStore.pop(claimId, None)
    # Generate new IDs (use the same helpers as the claimSubmitted reset path)
    claimId = str(uuid.uuid4())
    threadId = str(uuid.uuid4())
    request.session["claimId"] = claimId
    request.session["threadId"] = threadId
    # Send QueueRotationSignal to rebind SSE stream to the new thread ID
    # (use the same signal mechanism as the claimSubmitted path)
    await popQueue(oldThreadId)  # oldThreadId captured before the update above
    # (Do NOT clear chat history on the frontend — the cancel ack is visible.
    # The next uploaded receipt starts fresh on the new thread.)
```

**Why `priorState` in the POST preamble, not the SSE loop:** The existing `claimSubmitted` rotation uses this pattern because the SSE loop only runs `runGraph()` — it has no access to `finalState`. Detecting the flag on the next incoming POST (via `graph.aget_state`) gives the same semantics: state persisted by the graph is read before processing the user's next action, then the session is rotated before the new graph run begins. This also routes through `QueueRotationSignal` which rebinds the SSE stream to the new thread ID, preventing orphaned threads.
