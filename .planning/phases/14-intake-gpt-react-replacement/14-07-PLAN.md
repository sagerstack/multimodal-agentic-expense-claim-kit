**Part A — Multi-tool processing fix:**

Currently `applyToolResultsNode` reads only `toolMessages[-1]` (the most recent tool result). When the LLM emits a multi-tool turn (e.g. searchPolicies + convertCurrency in one AIMessage with two tool_calls), only the last result is processed, and the first is silently dropped.

Refactor `applyToolResultsNode` to iterate over ALL ToolMessages produced in the current turn, in order:

```python
async def applyToolResultsNode(state: IntakeGptGraphState) -> dict[str, Any]:
    messages = state["messages"]
    # Find all ToolMessages emitted since the last AIMessage with tool_calls
    # (i.e. the current turn's tool results)
    toolMessages = []
    for msg in reversed(messages):
        if isinstance(msg, ToolMessage):
            toolMessages.append(msg)
        elif isinstance(msg, AIMessage) and msg.tool_calls:
            break
    toolMessages.reverse()  # restore chronological order

    if not toolMessages:
        return {}

    # ... existing setup of intakeState, slots, updates ...

    # Process each tool result cumulatively
    for toolMessage in toolMessages:
        toolName = toolMessage.name
        # ... dispatch on toolName, mutating slots/intakeState in place ...

    updates["intakeGpt"] = intakeState
    return updates
```

**Important:** The requestHumanInput dispatch (which sets pendingInterrupt) MUST still see the correct ordering — if a turn somehow produced both a non-blocking tool result AND a requestHumanInput, the pendingInterrupt branch must be the FINAL state. The iteration order (chronological) preserves this because requestHumanInput is always the last tool in any turn that includes it (LLM is prompted to call it last; runtime synthetic AIMessages only ever contain a single tool_call).
