# Phase 6.1 Context: Model Upgrade + UX Fixes

## Problem Statement

Two issues identified during Phase 7 UAT:

### 1. QwQ-32B Reasoning Bottleneck

The current LLM (`qwen/qwq-32b`) is a reasoning model that generates extremely long chains of thought, dominating response time:

- **97% of turn time is QwQ reasoning**, only 3% is tool execution
- Average reasoning per turn: 1m 43s
- Worst case: 3m 30s for a single text response with 0 tools
- Tool execution: 0.8-5s per tool call
- Total session time: ~7 minutes across 4 turns

Root cause: QwQ-32B generates CoT as regular output tokens with no way to control reasoning length. No `reasoning_effort` or `reasoning.max_tokens` support.

### 2. Submission Summary Panel Not Updating After Claim Submission

After claim submission (CLAIM-XXXX confirmed), the Submission Summary panel:
- Still shows "50% Complete" instead of 100%
- Still shows "Current Session" header instead of the Claim ID
- Still shows "USD 98.56" without conversion to SGD
- Category shows "--" instead of actual category
- "Submit Entire Batch" button still visible after submission

The `_extractSummaryData` falls back to graph state but `claimSubmitted` flag detection may not work across the final turn when the submit tool runs in a prior interrupted turn.

## Decisions

### Model Switch
- **LLM**: Switch from `qwen/qwq-32b` to `qwen/qwen3-235b-a22b-2507`
  - 235B MoE with 22B active params = fast inference
  - No thinking mode (`<think>` blocks) = no long CoT chains
  - Optimized for tool usage
  - 262K context length
  - Cheaper: $0.071/$0.10 per 1M tokens
- **VLM**: Keep `google/gemini-2.0-flash-001` (already fast, 5s extraction)
- **Thinking panel**: Becomes a tool activity log (step names + summaries), no reasoning previews needed

### BUG-013: Hallucinated Claim Submission (Critical)
- QwQ-32B skipped `submitClaim` tool call entirely, generated fake "CLAIM-1523 successfully submitted" response
- 0 tools called in submission turn, claim never persisted to DB
- Mitigations needed at multiple levels:
  1. Model switch (better tool compliance)
  2. System prompt: "NEVER state submitted unless you received a claim number from submitClaim response"
  3. Post-turn server-side validation: if AI response mentions "submitted" but `submitClaim` not in thinkingEntries, inject error/warning
  4. Consider: evaluatorGate should only fire if `claimSubmitted` is set by the actual tool, not by the LLM's text

### System Prompt
- Trim redundant sections from `agentSystemPrompt.py`
- Remove tool docs (LangGraph provides via tool schemas)
- Condense phase instructions
- Remove XML tag warnings (new model doesn't emit them)

### Summary Panel Fixes
- After submission: show Claim ID instead of "Current Session"
- After submission: show 100% complete
- Show converted SGD amount and actual category
- Hide "Submit Entire Batch" button after submission
- Propagate `claimSubmitted` + claim number through graph state to summary panel
