## Summary

This phase has two orthogonal problems: a model swap and a UX bug fix. Both are well-scoped changes to existing code — no new dependencies, no new architecture.

**Problem 1: Model swap.** Replace `qwen/qwq-32b` with `qwen/qwen3-235b-a22b-2507` in `.env.local` and adjust the config and intake node for the new model's characteristics. The new model has no `<think>` blocks — the entire reasoning cleanup path in `sseHelpers.py` (`_stripThinkingTags`, `reasoningBuffer`, the "Reasoning..." step entries) can be removed or neutered. The system prompt must also be trimmed: the 22-line TOOLS section (lines 149-170) and the 3-line XML tag warning (lines 16-18) are the primary targets. Total reduction: ~50 lines, landing under 100.

**Problem 2: Summary panel bugs (BUG-012) + hallucination guard (BUG-013).** The panel has three root causes: (a) `claimNumber` is never propagated to `ClaimState` or the summary template — the `"Current Session"` label and `{{ claimNumber }}` variable don't exist yet; (b) `progressPct` reaches 100% only if `submitted=True` is correctly detected, but when `extractReceiptFields` runs in turn 1 and `submitClaim` in turn 3, the submission turn's `thinkingEntries` has no receipt data, so `hasReceiptData=False` and the whole function returns `None`; (c) the "Submit Entire Batch" button is hardcoded in `chat.html` with no conditional — it needs to be inside the `summary-update` SSE swap target. BUG-013 guard needs one server-side check: if the final AI message text contains "submitted" or "CLAIM-" but `submitClaim` is not in `thinkingEntries`, inject an error SSE event instead of the message.

**Primary recommendation:** Three independent work streams that can be planned as separate tasks: (1) model switch + prompt trim, (2) summary panel state propagation, (3) BUG-013 hallucination guard.

## Architecture Patterns

### Pattern 1: Model Config Update (one file change)

The model name lives in `.env.local` and is read by `getSettings()` → `openrouter_model_llm`. No code changes needed in the intake node — it already reads from `settings.openrouter_model_llm`.

**Files to change:**
- `.env.local` — line 17: `OPENROUTER_MODEL_LLM=qwen/qwq-32b` → `OPENROUTER_MODEL_LLM=qwen/qwen3-235b-a22b-2507`
- `.env.local` — also update `OPENROUTER_LLM_TEMPERATURE` (currently 0.3 — appropriate for instruction-following, keep or set to 0.6 per Qwen3 docs)

### Pattern 2: Reasoning Code Removal from sseHelpers.py

The new model produces zero `reasoning_content` in `additional_kwargs` and zero `<think>` blocks in token content. The reasoning-specific code paths become dead code but are harmless if left in. However, two behaviors actively cause problems with the new model if reasoning content is always empty:

1. The "Reasoning..." step name/content block (lines 440-453 and 464-474) — fires when `reasoningBuffer` is non-empty. With new model, `reasoningBuffer` will always be empty, so this block simply never fires. **No removal needed for correctness.**

2. `_stripThinkingTags` — applied to `finalText`. New model produces no tags, so the regex runs and matches nothing. Harmless. **No removal needed for correctness.**

3. `_stripToolCallJson` — QwQ-32B emitted raw JSON tool specs in text content alongside structured function calls. New model may not. Function is harmless if called on clean text (returns input unchanged). **No removal needed for correctness.**

**Conclusion:** The reasoning cleanup code does not need to be removed for Phase 6.1 to work correctly. Remove only if the CONTEXT.md explicitly requires it for cleanliness (it does say "no reasoning previews needed" but doesn't say remove the code).

### Pattern 3: System Prompt Trim — Sections to Remove

The current prompt is 170 lines. Target: under 100 lines.

### Pattern 4: Summary Panel — State Propagation Design

**Root cause analysis of BUG-012:**

The `_extractSummaryData` function in `sseHelpers.py` returns `None` when `hasReceiptData=False`. On the submission turn (turn 3), `thinkingEntries` contains `submitClaim` but NOT `extractReceiptFields` (that ran in turn 1). The function falls back to `graphState` but `graphState.get("extractedReceipt")` only works if the intake node propagates `extractedReceipt` back into `ClaimState` — which it currently does NOT (see `node.py` line 120: `stateUpdate = {"messages": result["messages"]}` — only messages are updated).

**Fix chain:**

1. **`intakeNode` in `node.py`**: After scanning tool messages for `submitClaim` success, also scan for `extractReceiptFields` result and populate `stateUpdate["extractedReceipt"]` + `stateUpdate["currencyConversion"]`. This ensures graph state has receipt data across all turns.

2. **`ClaimState` in `state.py`**: Add `claimNumber: Optional[str]` field. The `intakeNode` sets this when `submitClaim` succeeds.

3. **`_extractSummaryData` in `sseHelpers.py`**: Extract `claimNumber` from `submitClaim` tool output. Add it to the returned dict: `{"claimNumber": "CLAIM-XXX", ...}`. Also extract from `graphState.get("claimNumber")` as fallback.

4. **`summary_panel.html` template**: Replace hardcoded `"Current Session"` with `{{ claimNumber or "Current Session" }}`. Add `{% if claimNumber %}` guard for conditional content.

5. **`chat.html`**: Move "Submit Entire Batch" button inside `id="summaryContent"` (the `sse-swap` target). This ensures it gets replaced on `summary-update` events. The `summary_panel.html` partial then conditionally renders it only when `progressPct < 100`.

**Data flow for claimNumber:**

```
submitClaim tool output → {"claim": {"claim_number": "CLAIM-003", ...}}
  → sseHelpers._extractSummaryData() reads data["claim"]["claim_number"]
  → summaryData["claimNumber"] = "CLAIM-003"
  → template renders {{ claimNumber }} = "CLAIM-003"
```

AND (for persistence to next turn):

```
intakeNode scans ToolMessage for submitClaim
  → parses content["claim"]["claim_number"]
  → stateUpdate["claimNumber"] = "CLAIM-003"
  → graphState["claimNumber"] set in LangGraph checkpoint
  → next turn: _extractSummaryData reads graphState["claimNumber"]
```

### Pattern 5: BUG-013 Hallucination Guard

Post-turn server-side validation in `runGraph` after the final response is extracted:

```python
# After finalText is set, before yielding MESSAGE event:
submittedInText = "submitted" in finalText.lower() or "CLAIM-" in finalText
submitCallMade = any(e.get("name") == "submitClaim" for e in thinkingEntries if e.get("type") == "tool")

if submittedInText and not submitCallMade:
    # Hallucination detected — replace with error
    errorHtml = templates.get_template("partials/message_bubble.html").render(
        content="I encountered an issue submitting your claim. Please try again by typing 'submit'.",
        isAi=True,
        confidenceScores=None,
        violations=None,
    )
    yield ServerSentEvent(raw_data=errorHtml, event=SseEvent.MESSAGE)
    return
```

This guard fires only when both conditions are true: the LLM claimed submission but no tool call happened. False positive risk is low — legitimate submission turns always have `submitClaim` in `thinkingEntries` because `intakeNode` scans for it.

## Common Pitfalls

### Pitfall 1: Wrong Model Variant (Thinking vs Instruct)

**What goes wrong:** Using `qwen/qwen3-235b-a22b-thinking-2507` instead of `qwen/qwen3-235b-a22b-2507` restores the CoT problem — the thinking variant generates long `<think>` blocks.

**How to avoid:** The model ID must end in `-2507` (the instruct variant), NOT `-thinking-2507`. Verify with a test call: the response should not contain `<think>` prefixes.

### Pitfall 2: "Submit Entire Batch" Button Outside SSE Swap Target

**What goes wrong:** The button is currently in `chat.html` OUTSIDE `id="summaryContent"`. Even if `summary_panel.html` conditionally hides it, the static button in `chat.html` will remain visible after SSE swap.

**How to avoid:** Move the button into `summary_panel.html` (inside the SSE swap target). The `chat.html` initial state can show a disabled placeholder. On `summary-update`, the full panel including button is replaced.

**Current state:** `chat.html` line 182 has the button hardcoded outside `id="summaryContent"`. The `id="summaryContent"` `sse-swap` target ends at line 180 (closing `</div>`).

### Pitfall 3: Summary Panel Returns None on Submission Turn

**What goes wrong:** On the submission turn, `hasReceiptData=False` because `thinkingEntries` has no `extractReceiptFields` entry and `graphState["extractedReceipt"]` is None (never populated by `intakeNode`). The function returns `None`, no `summary-update` SSE event is emitted, panel stays at 50%.

**How to avoid:** Fix `intakeNode` to write `extractedReceipt` and `currencyConversion` into `stateUpdate` when those tool messages are present in `result["messages"]`. This is the prerequisite for the graphState fallback to work.

### Pitfall 4: progressPct Logic Already Correct, But Never Triggered

**What goes wrong:** The progressPct=100 branch (line 237 of sseHelpers.py) is correct code — but it requires `submitted=True`. `submitted` is only set to `True` if the submitClaim tool output is in `thinkingEntries` (current turn) OR `graphState["claimSubmitted"]` is True. The `claimSubmitted` flag IS set by `intakeNode` correctly (line 134 of node.py). The real bug is that `hasReceiptData=False` causes early return before `submitted=True` can matter.

**How to avoid:** Fix `hasReceiptData` first. Once graph state has `extractedReceipt`, the fallback path sets `hasReceiptData=True`, `submitted=True` (via `graphState["claimSubmitted"]`), and `progressPct=100`.

### Pitfall 5: Category Shows "--" on Submission Turn

**What goes wrong:** Category comes from `extractReceiptFields` output (`fields.get("category", "")`). On submission turn, this isn't in `thinkingEntries`. The graphState fallback reads `graphState["extractedReceipt"]["fields"]["category"]` — but this only works if `intakeNode` populated `extractedReceipt` in graph state (see Pitfall 3).

**Same fix:** Populating `extractedReceipt` in `stateUpdate` from `intakeNode` resolves both missing category and missing merchant on submission turn.

### Pitfall 6: claimNumber Propagation Requires Two Paths

**What goes wrong:** If only the `thinkingEntries` path is implemented (claim number from current turn's `submitClaim` output), and the user navigates away or triggers another turn, the panel reverts to "Current Session" because `thinkingEntries` is per-turn.

**How to avoid:** Implement both paths:
1. Current turn: extract from `thinkingEntries` submitClaim output
2. Subsequent turns: read from `graphState["claimNumber"]` (requires adding `claimNumber` to `ClaimState` and populating it in `intakeNode`)

elif toolName == "submitClaim":
    if "error" not in data:
        submitted = True
        claimData = data.get("claim", {})
        extractedClaimNumber = claimData.get("claim_number", "")

### intakeNode — Populate extractedReceipt in stateUpdate

```python
# In intakeNode, scan result["messages"] for tool outputs:
for msg in result["messages"]:
    if hasattr(msg, "name") and hasattr(msg, "content"):
        try:
            content = json.loads(msg.content) if isinstance(msg.content, str) else msg.content
            if isinstance(content, dict) and "error" not in content:
                if msg.name == "submitClaim":
                    stateUpdate["claimSubmitted"] = True
                    claimRecord = content.get("claim", {})
                    if claimRecord.get("claim_number"):
                        stateUpdate["claimNumber"] = claimRecord["claim_number"]
                elif msg.name == "extractReceiptFields":
                    stateUpdate["extractedReceipt"] = content
                elif msg.name == "convertCurrency":
                    stateUpdate["currencyConversion"] = content
        except (json.JSONDecodeError, TypeError):
            pass
```

   - What's unclear: Whether 0.3 causes the instruct model to be overly terse
   - Recommendation: Test with 0.6; if too verbose, drop back to 0.3

3. **Token streaming behavior with new model**
   - What we know: QwQ-32B streams reasoning tokens via `additional_kwargs["reasoning_content"]` and regular tokens via `content`. New model only streams regular content tokens.
   - What's unclear: Whether `on_chat_model_stream` events fire during tool call generation (before the actual tool call) with the new model, or only after
   - Recommendation: Keep `pendingToolCalls` counter and the `if pendingToolCalls > 0: continue` guard — it's model-agnostic and prevents showing intermediate tokens
