## Key context from summaries and UAT

### What exists now (from 02.2-02-SUMMARY):
- System prompt (agentSystemPrompt.py): 12-step strict checklist with "Step N:" user-facing output contract
- app.py message handling: Iterates ALL messages in result["messages"], sends every AI message with content
- app.py Step element: Sets processingStep.input/output to static strings like "Processing receipt image..." / "Receipt processing complete"

### UAT Test 3 failure (agent stalling + message duplication):
- Root cause 1 (stalling): Agent completed graph invoke (returned 6 messages) but only 2 passed the filter. Agent stopped after extraction because Step 3 (cross-reference) needs a user description, but app.py sends generic "uploaded receipt for claim {id}". Agent needs user input to continue but doesn't know how to skip.
- Root cause 2 (duplication): app.py sends ALL AI messages from result["messages"] — this is the FULL conversation history (all turns), not just new messages. On each user interaction, all prior AI messages are re-displayed.

### UAT Test 4 failure (CoT not captured):
- Root cause: processingStep.input and processingStep.output are set to static strings. ToolMessages, empty AI messages, and intermediate reasoning are filtered out and discarded — they should be captured as Step content.

### UAT Test 5 failure (conversational UX):
- Root cause: System prompt mandates "Step N:" format for user-facing output. The CONTEXT.md post-UAT redesign mandates a two-layer model:
  - Layer 1 (user-facing): Conversational assistant tone, no numbered steps, tables for data, 4-5 messages per receipt
  - Layer 2 (CoT/on-demand): Collapsible Thinking panel with full tool calls and reasoning detail

### CONTEXT.md conversational flow (what user should see for a clean receipt):
1. "Thinking..." indicator (collapsible CoT panel)
2. "Here's what I extracted from your receipt:" + markdown table
3. Pre-submission summary: full claim card + policy check results + "Ready to submit?"
4. Confirmation: Claim ID + next steps

### CONTEXT.md interaction rules:
- Autonomous by default — no pause unless reason to pause
- Cross-reference OPTIONAL — only if user provided description
- No pause after extraction — continue unless low-confidence fields
- Leave low-confidence fields blank and ask (no auto-fill)
- Policy violations: warn + allow override
- Conversation stays open until submission

### Current app.py line 148 — the generic message sent with image upload:
```python
humanMsg = HumanMessage(
    content=f"I've uploaded a receipt image for claim {claimId}. Please process it using extractReceiptFields."
)
```
This provides NO user description for cross-reference. The prompt should know to skip cross-reference in this case.

### Current app.py message filtering (lines 191-207):

Problem: `result["messages"]` contains ALL messages (entire conversation), not just new ones from this turn.

    **Fix 1: Message duplication — track message count before invoke, only send NEW messages**

    Before invoking the graph, count the existing messages in the conversation state so we know where new messages start:

    ```python
    # Before graph.ainvoke(), get current message count from graph state
    try:
        currentState = await graph.aget_state(config={"configurable": {"thread_id": threadId}})
        messageCountBefore = len(currentState.values.get("messages", []))
    except Exception:
        messageCountBefore = 0
    ```

    After invoke, when processing result["messages"], skip the first messageCountBefore messages:
    ```python
    agentMessages = result["messages"]
    newMessages = agentMessages[messageCountBefore:]  # Only new messages from this turn
    ```

    Then iterate over `newMessages` instead of `agentMessages`.

    IMPORTANT: For the FIRST invoke (image upload), there are no prior messages in state, so messageCountBefore will be 0 and all messages are "new". For subsequent invokes (clarification responses), messageCountBefore reflects the messages from prior turns.

    **Fix 2: CoT capture — write filtered-out messages into Step element**

    Currently, filtered-out messages (ToolMessages, empty AI messages, AI messages with tool_calls only) are silently discarded. Instead, collect them and write them as the Step's output content.

    Replace the static `processingStep.output = "Receipt processing complete"` with actual CoT content:

    ```python
    # Inside the Step context, after graph invoke and before exiting the Step:
    cotEntries = []
    userFacingMessages = []

    for msg in newMessages:
        if hasattr(msg, "type") and msg.type == "ai" and hasattr(msg, "content") and msg.content:
            # Check if this is a tool-call-only message (has tool_calls but content is just announcing the call)
            if hasattr(msg, "tool_calls") and msg.tool_calls and not msg.content.strip():
                cotEntries.append(f"**Tool Call:** {msg.tool_calls[0].get('name', 'unknown')}")
            else:
                userFacingMessages.append(msg)
        elif hasattr(msg, "type") and msg.type == "tool":
            # ToolMessage — capture for CoT
            toolName = getattr(msg, "name", "unknown")
            toolContent = str(msg.content)[:500] if hasattr(msg, "content") else ""
            cotEntries.append(f"**Tool Result ({toolName}):** {toolContent}")
        elif hasattr(msg, "type") and msg.type == "ai":
            # AI message with no content (tool_calls only) — capture for CoT
            if hasattr(msg, "tool_calls") and msg.tool_calls:
                for tc in msg.tool_calls:
                    cotEntries.append(f"**Tool Call:** {tc.get('name', 'unknown')}({json.dumps(tc.get('args', {}))[:200]})")

    # Write CoT to Step output
    if cotEntries:
        processingStep.output = "\n\n".join(cotEntries)
    else:
        processingStep.output = "No intermediate reasoning captured"
    ```

    Then, OUTSIDE the Step context (after the `async with cl.Step` block), send only the userFacingMessages to the user. This means moving the message-sending loop OUTSIDE the Step block.

    **Fix 3: Update the HumanMessage for image upload to include user's text if provided**

    ```python
        )
    ```

    This gives the agent the user's description (if any) for cross-referencing, and explicitly notes when no description exists so the agent knows to skip cross-reference per the prompt's CONDITIONAL instruction.

    **Structural change summary for onMessage:**

    The new structure of the message handling in onMessage should be:

    1. Get graph, threadId, claimId, awaitingClarification from session
    2. Process image attachment (existing logic)
    3. Get messageCountBefore from graph state
    4. `async with cl.Step(...)` block:
       a. Set processingStep.input

    - CoT content is written to the Step BEFORE exiting the `async with` block
    - User-facing messages are sent AFTER the Step closes (they appear in main chat, not inside the Step)
    - Only NEW messages from this turn are processed
    - The "Thinking..." step is visible while processing, then collapses
