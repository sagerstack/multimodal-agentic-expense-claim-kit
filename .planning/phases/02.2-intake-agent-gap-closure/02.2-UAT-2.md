## Gaps

### Issue A: submitClaim Fails — Field Name Mismatch (BLOCKER)
severity: critical
files: src/agentic_claims/agents/intake/tools/submitClaim.py, src/agentic_claims/agents/intake/prompts/agentSystemPrompt.py, mcp_servers/db/server.py
root_cause: |
  Field name mismatch between what the agent sends and what MCP insertClaim expects.
  The error: "14 validation errors for call[insertClaim]\nclaimNumber Missing required argument"

  Agent sends claimData with: claimantId, merchant, expenseDate, amountSgd, originalAmount,
  originalCurrency, category...

  MCP insertClaim requires:
  - claimNumber (str, REQUIRED) — agent never generates one
  - employeeId (str, REQUIRED) — agent sends "claimantId" instead
  - status (str, REQUIRED) — agent doesn't set this
  - totalAmount (float, REQUIRED) — agent sends "amountSgd" instead

  Additionally, submitClaim.py merge logic (line 40-44) prefixes all receipt fields
  with "receipt" — creates "receiptMerchant" but MCP expects "merchant".

  Fix requires:
  1. submitClaim tool must map agent field names to MCP field names (or generate missing fields like claimNumber, status)
  2. System prompt Step 4/11 should align field names with actual tool interface
  3. Merge logic should not blindly prefix receipt fields — some MCP fields don't use "receipt" prefix

### Issue B: Agent Doesn't Self-Diagnose Tool Errors
severity: high
files: src/agentic_claims/agents/intake/prompts/agentSystemPrompt.py
root_cause: |
  When submitClaim returns an error with specific details ("14 validation errors...
  claimNumber Missing required argument"), the agent has the error text in its context
  but asks the USER to identify the problem instead of parsing the error itself.

  The agent should: read the error, identify missing/wrong fields from the error message,
  and either retry with corrected fields or tell the user exactly what went wrong.

  Fix: Add error handling guidance to system prompt Step 11-12. When a tool returns an
  error, the agent should analyze the error message, diagnose the issue, and either
  self-correct or explain the specific problem to the user — never ask the user to
  debug a system error.

### Issue C: Policy Check Result Invisible to User
severity: high
files: src/agentic_claims/agents/intake/prompts/agentSystemPrompt.py
root_cause: |
  The agent runs searchPolicies and evaluates the claim against policy internally, but
  never produces a user-facing message about it. The user sees extraction table, then
  currency conversion, then jumps straight to "Everything looks good! Here's your claim
  summary:" — the policy check is swallowed silently.

  The prompt says to present policy results "conversationally" (Step 9) and embed in the
  summary card (Step 10), but "Be autonomous" + "No pause after extraction" causes the
  LLM to skip producing a separate policy check message. The user has no visibility into
  what was checked or why it passed.

  Fix: The prompt should explicitly require a user-facing message after policy check,
  e.g. "Your meal expense of SGD 20.72 is within the daily cap of SGD 100 (Section 2.1)."
  This is separate from the summary card — the user should SEE the policy check happen.

### Issue D: Currency Conversion Shows Placeholder Before Tool Result
severity: medium
files: src/agentic_claims/agents/intake/prompts/agentSystemPrompt.py
root_cause: |
  Agent outputs "Original: USD 16.20 → SGD [amount] (rate: [rate])" as a message BEFORE
  calling convertCurrency, then shows actual values after. The user sees broken template
  text. The prompt Step 5 example format ("Original: USD 50.00 → SGD 67.50 (rate: 1.35)")
  is interpreted as a template to fill incrementally.

  Fix: Prompt should instruct agent to call convertCurrency FIRST, then present the
  result in one message. Never output placeholder values.

### Issue E: Placeholder [claimantId] in Summary Table
severity: medium
files: src/agentic_claims/agents/intake/prompts/agentSystemPrompt.py
root_cause: |
  The pre-submission summary table shows literal "[claimantId]" instead of the actual UUID.
  The prompt's "Typical message flow" template (lines 28-36) uses bracket notation
  [claimantId] that the LLM outputs literally. The few-shot examples (line 140) use
  concrete values ("EMP-001") but the template section confuses the model.

  Fix: Remove bracket notation from template section, or use consistent concrete
  examples throughout. The LLM should never output square-bracket placeholders.

### Issue F: Third-Party Loggers Dump Unstructured Payloads to Seq
severity: low
files: src/agentic_claims/core/logging.py
root_cause: |
  setupLogging() attaches the SeqHandler at the root logger level, so ALL loggers
  (including third-party like openai._base_client) send output to Seq. The OpenAI SDK
  emits verbose debug logs with full HTTP request bodies (headers, system prompt, etc.)
  as unstructured text blobs.

  Fix: Either set third-party loggers (openai, httpx, httpcore) to WARNING level so
  they don't flood Seq, or attach SeqHandler only to the "agentic_claims" namespace.

### Issue G: Step Element Naming and Streaming CoT
severity: medium
files: src/agentic_claims/app.py
root_cause: |
  Two related UX issues with the Chainlit Step element:

  1. Step name shows "Used Processing Receipt" which is meaningless to the user.
     Should show "Thinking" while busy, then "Thought for X min Y seconds" when done
     (similar to ChatGPT/Claude thinking UI).

  2. CoT content only appears AFTER the entire agent loop finishes. While the LLM is
     processing, the Step panel is empty — user has no visibility into what's happening.
     The Step should populate in real-time as tool calls and results stream in.

  Fix for naming: Set Step name="Thinking", track start time, update name to
  "Thought for {duration}" after completion.

  Fix for streaming: Switch from graph.ainvoke() to graph.astream_events() in app.py.
  As each event arrives (tool call, tool result), append to processingStep.output
  incrementally via processingStep.stream_token() or output updates. Requires
  restructuring the invoke/message-handling flow.

### Issue H: No Model Fallback on OpenRouter 402 (Quota Exhausted)
severity: high
files: src/agentic_claims/agents/intake/tools/extractReceiptFields.py, src/agentic_claims/core/config.py
root_cause: |
  When OpenRouter returns HTTP 402 ("This request requires more credits, or fewer
  max_tokens. You requested up to 27621 tokens, but can only afford 25188"), the
  tool fails and the agent tells the user to "try re-uploading later" — the workflow
  is dead.

  The system should automatically retry with a fallback model (cheaper/smaller) when
  the primary model's quota is exhausted. This applies to any tool that calls
  OpenRouter (extractReceiptFields uses the VLM model, agent loop uses the chat model).

  Fix: Add fallback model config (e.g., FALLBACK_VLM_MODEL, FALLBACK_CHAT_MODEL in
  settings). On 402 response, retry the same request with the fallback model. Log the
  fallback so it's visible in Seq.
