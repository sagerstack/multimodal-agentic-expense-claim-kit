    Replace the entire `INTAKE_AGENT_SYSTEM_PROMPT` with the 12-step strict checklist prompt. The prompt structure must follow this exact layout (from CONTEXT.md decisions):

    **Section 1: SUCCESS CRITERIA** (at the TOP — most important, LLM focuses on first content)
    - Process every receipt through all 12 steps in exact order
    - Send ONE message per step (not consolidated)
    - Always announce action BEFORE executing it
    - Numeric comparisons must be mathematically correct: "IMPORTANT: 98.56 < 100 means NO violation. Always compare numbers correctly."
    - Flag contradictions between user description and receipt data
    - Receipt data ALWAYS takes precedence over user description

    **Section 2: OUTPUT CONTRACT**
    - Format: "Step N: [Action]" -> execute -> show result -> announce next step
    - Use markdown tables for structured data (extracted fields, policy clauses, confirmation summary)
    - Chain-of-thought reasoning goes in tool calls, NOT in user-facing messages
    - Each step is a separate message (never combine multiple steps)

    **Section 3: 12-STEP WORKFLOW** (strict numbered checklist — see CONTEXT.md for all 12 steps)
    For each step, include:
    - Step number and name
    - What to do (1-2 sentences)
    - Output format (what the message should look like)
    - Few-shot example for complex steps ONLY (steps 2, 3, 9, 10 — the ones with tables or non-obvious format). Steps 5 and 8 are straightforward enough without examples.

    Steps in order:
    1. Receipt quality check — validate image, output pass/fail with metrics
    2. Extract fields — call VLM, show table with all fields and confidence scores (include few-shot table example)
    3. Cross-reference description vs receipt — compare user's stated expense against extracted data using full conversation context. Flag category mismatches (user said "hotel" but receipt is restaurant), amount discrepancies, date inconsistencies. Receipt data takes precedence. If mismatch, ask user to clarify. If user confirms despite mismatch, accept but flag in intakeFindings. (include few-shot mismatch example)
    4. Prepare claim — fill mandatory submission attributes from extracted data
    5. Currency conversion — if non-SGD, convert and show original + converted + rate
    6. Identify gaps + low confidence — flag missing mandatory data and uncertain extractions (confidence < threshold) together
    7. Clarifications — ask user to confirm/correct ALL flagged items in one pass (not one at a time)
    8. Check policy — search policies via RAG, show table of relevant clauses with section references
    9. Flag violations — evaluate claim against policies. CRITICAL: Compare numbers correctly. Show the comparison explicitly ("Claim: SGD 98.56, Limit: SGD 100, Result: 98.56 < 100 = NO violation"). If violations exist, ask for justification. (include few-shot correct evaluation example)
    10. Confirm — show finalized claim summary card as table with all mandatory fields, ask for explicit "yes" or "confirm" (include few-shot summary table example)
    11. Submit — call submitClaim tool with claimData, receiptData, and intakeFindings
    12. Provide claim ID — show claim ID from submission result

    **Section 4: TOOLS** (brief list with signatures)
    - extractReceiptFields(claimId: str) -> dict
    - searchPolicies(query: str) -> list
    - convertCurrency(amount: float, fromCurrency: str) -> dict
    - submitClaim(claimData: dict, receiptData: dict, intakeFindings: dict) -> dict
    - askHuman(question: str, data: dict) -> dict

    **Section 5: CONSTRAINTS** (at the END — reinforcement)
    - Never skip steps, even if data seems complete
    - Never combine multiple steps into one message
    - Never make assumptions about missing data — always ask
    - Receipt data ALWAYS takes precedence over user description
    - If user confirms despite mismatch, accept but flag in intakeFindings for reviewer
    - Numeric policy evaluation must show explicit comparison (X vs Y = result)
    - intakeFindings must accumulate throughout the workflow: mismatches found in step 3, overrides in step 7, violations in step 9

    **Prompt length target:** Keep under 3000 tokens. Use few-shot examples only for steps with tables or non-obvious formats (steps 2, 3, 9, 10 only — not every step). Success criteria and output contract at the TOP before step definitions.

    **Update tests:** In `tests/test_intake_agent.py`, update any test that checks prompt content to match the new structure (e.g., if a test asserts "Workflow:" is in the prompt, update to assert "12-STEP WORKFLOW" or "SUCCESS CRITERIA" is present).

  <files>

  <action>
    Rewrite the message handling in `onMessage()` to:

    1. **Wrap agent invocation in a Chainlit Step** for chain-of-thought containment:
       - Use `async with cl.Step(name="Processing Receipt", type="tool") as processingStep:` to wrap the `graph.ainvoke()` call
       - Set `processingStep.input` to describe what's happening ("Processing receipt image..." or "Handling user response...")
       - Set `processingStep.output` to a brief summary when done
       - This makes all intermediate agent reasoning (tool calls, internal messages) collapsible in the UI

    2. **Filter agent output messages**: When extracting messages from the result:
       - Only send AI messages that have non-empty `content` (skip ToolMessages, function calls, empty AI messages)
       - Skip messages where `msg.type == "tool"` (these are tool call results, not user-facing)
       - For AI messages with content, send each as a separate `cl.Message`
       - This ensures the user only sees the agent's step-by-step announcements, not raw tool reasoning

    3. **Add structured logging to Chainlit app**:
       - Import `logging` and create logger at module level: `logger = logging.getLogger(__name__)`
       - Log at session start: "Chat session started", thread_id, claim_id
       - Log at message received: message content length, has_image, awaiting_clarification
       - Log before graph invoke: input state keys
       - Log after graph invoke: result type, has_interrupt, message count
       - Log at message send: each user-facing message (first 100 chars)

    4. **Initialize logging in app startup**: In `onChatStart()`, call `setupLogging()` from `agentic_claims.core.logging` (import it). Use a try/except so logging setup failure doesn't crash the app — log to stderr as fallback.

    5. **Keep the existing interrupt/resume flow intact** — don't change the clarification logic, just wrap it in the Step context manager and add logging.

    **Important Chainlit Step configuration note**: Check if `chainlit.toml` or `.chainlit/config.toml` exists. If so, ensure `[UI]` section has `cot = "tool_calls"` (show Steps but collapsed). If no config file exists, leave it — Chainlit's default behavior shows Steps collapsed.
