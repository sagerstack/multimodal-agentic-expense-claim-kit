<decisions>
## Implementation Decisions

### UX Paradigm: Two-Layer Conversational Model

**Fundamental shift from Phase 2.1**: The 12-step numbered checklist becomes INTERNAL agent reasoning (CoT), NOT user-facing output. The user sees a conversational experience with optional CoT visibility.

**Layer 1 — Conversational Flow (user-facing)**:
- Professional assistant tone
- 4-5 messages per receipt processing (brief transitions between actions)
- No numbered steps visible — agent speaks conversationally ("Here's what I extracted:", "Everything checks out against policy. Ready to submit?")
- Extracted data shown as a table embedded in a conversational message (not as a numbered step)
- Conflicts/issues surfaced conversationally with discrete options for user to choose from

**Layer 2 — Chain-of-Thought (on-demand)**:
- Collapsible "Thinking..." indicator with elapsed time (ChatGPT-style) visible while agent processes
- Expands to show full tool calls and LLM reasoning (developer-level transparency)
- All 12 internal steps logged here with complete detail
- Non-blocking issues logged in CoT AND mentioned conversationally (both layers)
- System errors: friendly message in conversation, full technical detail in CoT

### Internal Processing Steps (CoT-only, not user-facing)

The agent still follows this sequence internally, but it is reasoning guidance — never shown to the user as "Step N:".

1. **Receipt quality check** — validate image resolution and blur before processing
2. **Extract fields from receipt** — call VLM extraction with confidence scores
3. **Cross-reference user description vs receipt** — OPTIONAL: only if user provided a description. No description = use receipt as sole source and skip
4. **Prepare claim** — fill all mandatory submission attributes from extracted data
5. **Currency conversion** — if non-SGD, convert and show both amounts
6. **Identify gaps + low confidence fields** — flag missing mandatory data and uncertain extractions
7. **Clarifications** — ask user to confirm/correct all flagged items in one pass
8. **Check policy** — search policies via RAG for relevant clauses
9. **Flag violations + ask for justification** — present violations with cited clauses
10. **Confirm** — show finalized claim summary card, ask for explicit confirmation
11. **Submit** — persist claim + receipt to database
12. **Provide claim ID** — show the created claim ID with next steps

### Conversational Flow (what the user actually sees)

Approximate message sequence for a clean receipt (no issues):

1. **Processing indicator**: "Thinking..." with timer (collapsible CoT panel active)
2. **Extraction result**: "Here's what I extracted from your receipt:" + table (vendor, amount, date, category, etc.)
3. **Pre-submission summary**: Full claim summary table with all fields, policy check results, converted amounts. "Ready to submit?"
4. **Confirmation response**: Claim ID + what happens next ("Claim EC-0042 submitted. Your manager will review within 48 hours.")

Additional messages inserted for:
- Low-confidence fields: leave blank and ask user directly ("I couldn't read the date clearly. What date was this receipt?")
- Policy violations: warn and allow override — flag clearly but let user submit with justification, logged for reviewer
- Description conflicts: explain the conflict, present 2-3 options for user to choose from
- Image quality failure: explain what's wrong, give guidance, ask for re-upload (conversation stays open)
- System errors: friendly retry prompt to user, technical detail in CoT

### Interaction Points

- **Autonomous by default**: Agent processes everything without pausing unless there's a reason to pause
- **Pause triggers**: (1) Before submission — always, (2) Conflict/ambiguity requiring user decision
- **No pause after extraction**: Agent doesn't stop to ask "does this look right?" unless there are low-confidence fields
- **Conversation stays open**: Until claim is submitted, user can always re-upload or correct. No dead ends.

### Mismatch Handling Behavior

- **Warn and present options (soft gate)**: Agent flags mismatch, explains the conflict, presents 2-3 discrete options for user to choose from
- **Receipt takes precedence**: Receipt data is the default over user description
- **Cross-reference is OPTIONAL**: Only runs if user provided an expense description. No description = skip entirely, use receipt as source
- **Detection scope**: Use full conversation context (all user messages) when cross-referencing
- **Override behavior**: If user confirms despite mismatch, accept but flag for reviewer visibility in intakeFindings

### Low-Confidence Field Handling

- **Leave blank and ask**: When extraction confidence is low, do NOT auto-fill with best guess. Leave the field empty and ask the user directly.
- **No threshold-based auto-fill**: Clean data over convenience. Every uncertain field gets user confirmation.

### Policy Violation Handling

- **Warn + allow override**: Agent flags violations with cited policy clauses but does NOT block submission
- **User provides justification**: If user wants to proceed, they provide a reason which is logged in intakeFindings
- **Reviewer sees everything**: Violations, overrides, and justifications all persisted for reviewer

### Error Handling

- **Image quality failure**: Explain issue + guidance + ask for re-upload. Conversation continues.
- **System errors**: Friendly message to user ("I ran into a technical issue. Would you like me to try again?"), full technical detail (service name, error, stack trace) in CoT panel
- **Non-blocking issues**: Surface conversationally AND log in CoT (both layers)

### Seq Log Ingestion (Bug Fix)

- **Root cause**: `core/logging.py` outputs JSON to stdout only — no HTTP handler ships logs to Seq ingestion API
- **Also**: `app.py` defines its own inline `setupLogging()` instead of importing from `core/logging.py`
- **Fix**: Add HTTP handler to `core/logging.py` that POSTs CLEF events to Seq when `seq_url` is set. Remove inline `setupLogging` from `app.py`.

### Message Duplication (Bug Fix)

- **Root cause**: `app.py` sends ALL AI messages from `result["messages"]` (full conversation history) on every turn
- **Fix**: Track message count before graph invoke, only send NEW messages after invoke completes

### CoT Capture in Chainlit Step (Bug Fix)

- **Root cause**: `app.py` sets `processingStep.input/output` to static strings. Filtered-out messages (ToolMessages, empty AI) are discarded instead of captured.
- **Fix**: Capture filtered-out messages (ToolMessages, intermediate AI reasoning) as Step output or sub-steps within the Chainlit Step element

### System Prompt Restructuring

- **Internal reasoning, not user-facing format**: System prompt instructs the LLM to follow 12 steps as internal reasoning, but output conversationally to the user
- **No "Step N:" in user output**: Steps are CoT. User sees natural language summaries.
- **Few-shot examples**: Show conversational output format, not numbered step format
- **Numeric policy evaluation guidance**: Explicit instruction for correct number comparison
- **Cross-reference conditional**: "If the user provided an expense description, cross-reference it with receipt data. If no description was provided, skip this step and use receipt data as the sole source."

### Intake Findings Persistence

- **Both ClaimState + Database**: Agent observations stored in `intakeFindings` (ClaimState) and `intake_findings` JSONB column (DB)
- **Includes**: mismatches, overrides, low-confidence flags, user justifications, policy violations, red flags
- **submitClaim passes findings**: The tool includes intakeFindings in the DB insert

### Startup Script

- **Location**: `scripts/startup.sh`
- **Always clean restart**: Script stops all active instances before starting
- **Scope**: docker compose up, health checks, Alembic migrations, RAG policy ingestion, Seq container
- **Reset mode**: `--reset` flag wipes DB volumes
- **Seq included**: Part of standard startup sequence

### Claude's Discretion

- Message count tracking implementation in app.py
- Few-shot example content in system prompt
- intakeFindings dict structure
- Health check implementation details

<specifics>
## Specific Ideas

- Agent output should feel like talking to a professional assistant, not watching a checklist execute
- Two levels of detail: conversational flow (visible) + thinking process (expandable on demand)
- "Thinking..." indicator with elapsed time — like ChatGPT's "Thought for Xs" panel
- Receipt data takes precedence over user description in all conflicts
- Startup script is the single entry point for new developers
- Confirmation shows full claim summary table — complete picture before committing
- Post-submission shows claim ID + what happens next (reviewer timeline)

</specifics>
