<decisions>
## Implementation Decisions

### Agent Output Structure

- **Separate messages per step**: Each processing step sends its own Chainlit message (not one consolidated response)
- **Always announce next step**: Agent explicitly states what it will do before doing it (e.g., "Next, I'll check this against expense policies...")
- **Full details per step**: Each step shows complete results (full field table, all policy clauses, conversion details)
- **Chain-of-thought in Chainlit Step elements**: Use Chainlit's native `cl.Step()` for collapsible CoT — hidden by default, expandable for transparency
- **Error display**: Warning-styled banner with retry option when a step fails (e.g., policy search down, currency API unreachable)
- **Confirmation format**: Structured summary card based on claim submission attributes (not all extracted receipt fields) — table/card with mandatory fields before submit

### 12-Step Intake Flow (Strict Numbered Checklist)

The system prompt must enforce this exact sequence. LLM follows steps 1-12 in order, announcing each.

1. **Receipt quality check** — validate image resolution and blur before processing
2. **Extract fields from receipt** — call VLM extraction, show full results with confidence scores
3. **Cross-reference user description vs receipt** — flag inconsistencies (category, amount, date); receipt data takes precedence; use full conversation context
4. **Prepare claim** — fill all mandatory submission attributes from extracted data
5. **Currency conversion** — if non-SGD, convert and show both amounts
6. **Identify gaps + low confidence fields** — flag missing mandatory data and uncertain extractions together
7. **Clarifications** — ask user to confirm/correct all flagged items in one pass
8. **Check policy** — search policies via RAG for relevant clauses
9. **Flag violations + ask for justification** — present violations with cited clauses, ask user if needed
10. **Confirm** — show finalized claim summary card with all mandatory fields, ask for explicit confirmation
11. **Submit** — persist claim + receipt to database
12. **Provide claim ID** — show the created claim ID to the user

### System Prompt Restructuring

- **Strict numbered checklist**: LLM must follow the 12 steps in exact order
- **Few-shot examples included**: Each step gets an example of what its output message should look like
- **Numeric policy evaluation guidance**: Explicit instruction that the LLM must correctly compare numbers (e.g., "98.56 < 100 means NO violation")
- **Cross-reference instruction**: Generic instruction to use LLM judgment to flag inconsistencies between user description and receipt data — not hard-coded rules for specific mismatch types

### Mismatch Handling Behavior

- **Warn and ask (soft gate)**: Agent flags mismatch, asks user to clarify, but receipt data takes precedence over user description
- **Detection scope**: Use full conversation context (all user messages) when cross-referencing, not just the initial message
- **Override behavior**: If user confirms despite mismatch, accept but flag for reviewer visibility in the claim record
- **Generic detection**: Instruct LLM to use judgment for inconsistencies (category, amount, date, or any other discrepancy) — no hard-coded mismatch types

### Intake Findings Persistence

- **Both ClaimState + Database**: Agent observations (mismatches, overrides, low-confidence flags, user justifications, red flags) stored in:
  - `intakeFindings` field on ClaimState (dict) — for downstream agents (Compliance, Fraud, Advisor)
  - `intake_findings` JSONB column on claims table — for reviewer and audit trail
- **New Alembic migration (003)**: Add `intake_findings` JSONB column to claims table
- **submitClaim passes findings**: The tool includes intakeFindings in the DB insert

### submitClaim Fix Strategy

- **Structured logging first**: Add Python logging with JSON formatter throughout the LangGraph pipeline — each node logs its actions, try/catch blocks capture errors/warnings
- **Seq dashboard**: New Docker container (datalust/seq) on localhost:5341 for structured log monitoring, dev-only access
- **Log file path from .env**: Configurable log output path
- **Merge insertReceipt into insertClaim**: Single atomic DB operation creates both claim + receipt (they're always persisted together after intake completes). Remove the separate insertReceipt call from submitClaim tool.
- **Debug the runtime issue**: After logging is in place, trace exactly where dict-to-string conversion happens in the Chainlit → LangGraph → MCP pipeline

### Startup Script

- **Location**: `scripts/startup.sh`
- **Always clean restart**: Script stops all active instances (`docker compose down`) before starting, every time
- **Scope**: docker compose up --build, wait for health checks (fixed 120s timeout), Alembic migrations, RAG policy ingestion, Seq container
- **Reset mode**: `--reset` flag wipes DB volumes, re-runs migrations, re-ingests policies from scratch
- **Colored output**: Each step shows colored status messages (green checkmarks on success, red X on failure)
- **Seq included**: Seq container is part of the standard startup sequence
