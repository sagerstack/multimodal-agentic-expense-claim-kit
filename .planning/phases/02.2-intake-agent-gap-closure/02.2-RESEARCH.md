**Key technical areas:**

1. **Structured logging with Seq** — Add Python JSON logging throughout the LangGraph pipeline, deploy Seq Docker container (datalust/seq) for centralized log viewing, trace the submitClaim dict-to-string bug through the Chainlit → LangGraph → MCP call stack.

2. **Chainlit Step elements** — Use `cl.Step()` context manager for collapsible chain-of-thought display. Each processing step (VLM extraction, policy search, currency conversion) gets its own Step with input/output, hidden by default but expandable for transparency.

3. **Agent output restructuring** — System prompt must enforce strict 12-step checklist with few-shot examples. Agent sends separate Chainlit messages per step (not one consolidated response), always announces next action before executing it.

4. **Startup script** — `scripts/startup.sh` runs docker compose up --build, waits for health checks (120s timeout), runs Alembic migrations, runs RAG ingestion, starts Seq container. `--reset` flag wipes volumes and re-ingests policies from scratch.

5. **Alembic JSONB migration** — Add `intake_findings` JSONB column to claims table using `postgresql.JSONB()` with server_default='{}'. Store agent observations (mismatches, overrides, red flags) for downstream agents and reviewer audit trail.

**Primary recommendation:** Start with structured logging + Seq deployment to debug the submitClaim issue. Once that's fixed, restructure system prompt with strict 12-step flow and few-shot examples. Add Chainlit Step elements for each major processing stage. Build startup script last (integrates all pieces).

### Pattern 1: Python JSON Logging with Seq

**What:** Configure Python's standard logging to output JSON, send to Seq via GELF or HTTP ingestion.

**When to use:** When you need centralized structured logging across microservices for debugging production issues.

**Example:**
```python
# Source: python-json-logger documentation + Seq docs
# src/agentic_claims/core/logging.py
import logging
import sys
from pythonjsonlogger.json import JsonFormatter
from agentic_claims.core.config import getSettings

def setupLogging():
    """Configure JSON logging with Seq ingestion."""
    settings = getSettings()

    # JSON formatter with custom fields
    formatter = JsonFormatter(
        fmt="%(asctime)s %(levelname)s %(name)s %(funcName)s %(lineno)s %(message)s",
        rename_fields={
            "asctime": "timestamp",
            "levelname": "level",
            "funcName": "function",
            "lineno": "line"
        },
        static_fields={
            "service": "agentic-claims",
            "environment": settings.environment,
        }
    )

    # Console handler (JSON output)
    consoleHandler = logging.StreamHandler(sys.stdout)
    consoleHandler.setFormatter(formatter)
    consoleHandler.setLevel(logging.DEBUG)

    # File handler (optional, for persistence)
    if settings.log_file_path:
        fileHandler = logging.FileHandler(settings.log_file_path)
        fileHandler.setFormatter(formatter)
        fileHandler.setLevel(logging.DEBUG)

    # Root logger configuration
    rootLogger = logging.getLogger()
    rootLogger.setLevel(logging.DEBUG)
    rootLogger.addHandler(consoleHandler)
    if settings.log_file_path:
        rootLogger.addHandler(fileHandler)

    logging.info("Structured logging initialized", extra={
        "seq_url": settings.seq_url,
        "log_level": logging.getLevelName(rootLogger.level)
    })

# Usage in tools/submitClaim.py
import logging

logger = logging.getLogger(__name__)

async def submitClaim(claimData: dict) -> dict:
    """Submit claim with structured logging."""
    logger.info("submitClaim called", extra={
        "claim_data_keys": list(claimData.keys()),
        "claim_id": claimData.get("claimId")
    })

    try:
        # Call MCP tool
        result = await mcpCallTool(...)

        logger.info("MCP tool call succeeded", extra={
            "result_type": type(result).__name__,
            "result_keys": list(result.keys()) if isinstance(result, dict) else None
        })

        return result

    except Exception as e:
        logger.error("submitClaim failed", extra={
            "error": str(e),
            "error_type": type(e).__name__
        }, exc_info=True)
        raise
```

### Pattern 2: Chainlit Step for Collapsible Chain-of-Thought

**What:** Use `cl.Step()` context manager to create hierarchical, collapsible steps in Chainlit chat UI.

**When to use:** When agent performs multi-step reasoning and you want to show progress without overwhelming the user.

**Example:**
```python
# Source: Chainlit documentation - Step API
import chainlit as cl

@cl.on_message
async def onMessage(message: cl.Message):
    """Process message with structured steps."""

    # Step 1: Receipt quality check
    async with cl.Step(name="Receipt Quality Check", type="tool") as qualityStep:
        qualityStep.input = "Analyzing image resolution and blur..."

        # Run quality check
        isAcceptable, variance = checkImageQuality(imageBytes)

        qualityStep.output = f"✓ Image quality acceptable (variance: {variance:.2f})"

    # Step 2: VLM extraction
    async with cl.Step(name="VLM Extraction", type="llm") as extractionStep:
        extractionStep.input = "Extracting receipt fields with confidence scores..."

        # Call VLM
        extracted = await extractReceiptFields(imageBytes)

        # Format output as table
        extractionStep.output = formatExtractedFields(extracted)

    # Step 3: Policy search (nested substeps)
    async with cl.Step(name="Policy Validation", type="tool") as policyStep:
        policyStep.input = "Searching expense policies for relevant rules..."

        async with cl.Step(name="RAG Search") as searchStep:
            searchStep.input = "Query: meal expenses daily limit"
            results = await searchPolicies("meal expenses daily limit")
            searchStep.output = f"Found {len(results)} relevant clauses"

        async with cl.Step(name="Policy Evaluation") as evalStep:
            evalStep.input = "Comparing claim against retrieved policies..."
            violations = evaluatePolicies(extracted, results)
            evalStep.output = formatViolations(violations)

        policyStep.output = "✓ Policy validation complete" if not violations else f"⚠ {len(violations)} violations found"

    # Final message (user-facing summary)
    await cl.Message(content="Receipt processed successfully. Ready for confirmation.").send()
```

**Configuration in chainlit.toml:**
```toml
[UI]
# Control step visibility
# "full" = show all steps expanded
# "hidden" = hide all steps (only final messages visible)
# "tool_calls" = show only tool call steps
cot = "tool_calls"  # Default: show steps but collapsed
```

**Key insight:** Steps are collapsible by default. User sees clean final messages, but can expand steps to debug or understand agent reasoning. Use `type="tool"` for tool calls, `type="llm"` for model interactions.

### Pattern 3: Bash Startup Script with Health Checks

**What:** Orchestration script that starts Docker Compose, waits for all services to be healthy, runs migrations, ingests policies.

**When to use:** When you need a single entry point for local development that handles complex startup dependencies.

**Example:**
```bash
# Source: Docker Compose health check examples + bash best practices
#!/bin/bash
# scripts/startup.sh

set -e  # Exit on error

# Colors for output
RED='\033[0;31m'
GREEN='\033[0;32m'
YELLOW='\033[1;33m'
NC='\033[0m'  # No color

# Configuration
TIMEOUT=120
CHECK_INTERVAL=5
RESET_MODE=false

# Parse arguments
while [[ $# -gt 0 ]]; do
    case $1 in
        --reset)
            RESET_MODE=true
            shift
            ;;
        *)
            echo "Unknown option: $1"
            exit 1
            ;;
    esac
done

echo -e "${GREEN}========================================${NC}"
echo -e "${GREEN}Agentic Claims Startup Script${NC}"
echo -e "${GREEN}========================================${NC}"

# Step 1: Stop existing instances
echo -e "\n${YELLOW}[1/6] Stopping existing containers...${NC}"
docker compose down

# Step 2: Reset volumes (if --reset flag)
if [ "$RESET_MODE" = true ]; then
    echo -e "\n${YELLOW}[2/6] Resetting volumes...${NC}"
    docker compose down -v
    echo -e "${GREEN}✓ Volumes wiped${NC}"
else
    echo -e "\n${YELLOW}[2/6] Keeping existing volumes${NC}"
fi

# Step 3: Start services
echo -e "\n${YELLOW}[3/6] Starting Docker Compose services...${NC}"
docker compose up -d --build

# Step 4: Wait for health checks
echo -e "\n${YELLOW}[4/6] Waiting for services to be healthy (timeout: ${TIMEOUT}s)...${NC}"

waitForHealthy() {
    local service=$1
    local elapsed=0

    while [ $elapsed -lt $TIMEOUT ]; do
        status=$(docker inspect --format='{{.State.Health.Status}}' $(docker compose ps -q $service) 2>/dev/null || echo "starting")

        if [ "$status" = "healthy" ]; then
            echo -e "${GREEN}  ✓ $service is healthy${NC}"
            return 0
        fi

        echo -ne "\r  ⏳ $service: $status (${elapsed}s elapsed)..."
        sleep $CHECK_INTERVAL
        elapsed=$((elapsed + CHECK_INTERVAL))
    done

    echo -e "\n${RED}  ✗ $service failed health check (timeout)${NC}"
    return 1
}

# Wait for each service
for service in postgres qdrant mcp-rag mcp-db mcp-currency mcp-email seq app; do
    if ! waitForHealthy $service; then
        echo -e "\n${RED}Startup failed. Check logs with: docker compose logs $service${NC}"
        exit 1
    fi
done

# Step 5: Run Alembic migrations
echo -e "\n${YELLOW}[5/6] Running database migrations...${NC}"
docker compose exec -T app alembic upgrade head
echo -e "${GREEN}✓ Migrations complete${NC}"

# Step 6: Ingest policies (if reset or empty)
echo -e "\n${YELLOW}[6/6] Ingesting policies...${NC}"
if [ "$RESET_MODE" = true ]; then
    docker compose exec -T app python scripts/ingest_policies.py --force
else
    docker compose exec -T app python scripts/ingest_policies.py --skip-if-exists
fi
echo -e "${GREEN}✓ Policies ingested${NC}"

# Success message
echo -e "\n${GREEN}========================================${NC}"
echo -e "${GREEN}✓ System ready${NC}"
echo -e "${GREEN}========================================${NC}"
echo -e "\nChainlit UI:      http://localhost:8000"
echo -e "Seq Logs:         http://localhost:5341"
echo -e "Postgres:         localhost:5432"
echo -e "Qdrant:           http://localhost:6333"
echo -e "\nPress Ctrl+C to stop services"

# Follow logs (optional)
docker compose logs -f app
```

**Key insight:** Health checks with timeouts prevent race conditions. Colored output makes status clear. `--reset` flag useful for testing clean deployments.

### Pattern 4: LLM Prompt with Strict Step Checklist

**What:** System prompt that enforces a numbered checklist workflow with explicit step announcements and few-shot examples.

**When to use:** When agent must follow a precise multi-step process and you need consistent, structured output.

**Example:**
```python
# Source: 2026 prompt engineering best practices
# src/agentic_claims/agents/intake/prompts/agentSystemPrompt.py

AGENT_SYSTEM_PROMPT = """You are an expense claims intake assistant. Your job is to process receipt images through a strict 12-step workflow.

## SUCCESS CRITERIA
- Every receipt processed through all 12 steps in exact order
- User receives clear, structured updates at each step
- No steps skipped, even if data seems obvious
- Numeric comparisons are mathematically correct (98.56 < 100 means NO violation)
- Contradictions between user description and receipt data are flagged

## OUTPUT CONTRACT
- Send ONE message per step (not one consolidated response)
- Always announce what you're doing BEFORE you do it
- Format: "Step N: [Action]" → perform action → show result → state next step
- Use tables for structured data (extracted fields, policy clauses)
- Chain-of-thought should be in tool reasoning, not user messages

## 12-STEP WORKFLOW (STRICT ORDER)

### Step 1: Receipt Quality Check
**What to do:** Validate image resolution and blur before processing
**Output format:** "Step 1: Checking receipt image quality... [result: ✓ acceptable / ✗ rejected]"
**Example:**
```
Step 1: Checking receipt image quality...

Resolution: 1280x1024 ✓
Blur score: 245.6 ✓

Image quality is acceptable. Proceeding to extraction.
```

### Step 2: Extract Fields from Receipt
**What to do:** Call VLM extraction tool, show full results with confidence scores
**Output format:** Table with all extracted fields and confidence
**Example:**
```
Step 2: Extracting fields from receipt...

| Field          | Value              | Confidence |
|----------------|-------------------|------------|
| Merchant       | DIG Restaurant    | 0.98       |
| Date           | 2026-03-24        | 0.95       |
| Total Amount   | 98.56             | 0.99       |
| Currency       | SGD               | 1.00       |
| Payment Method | VISA ****1234     | 0.92       |

Extraction complete.
```

### Step 3: Cross-Reference User Description vs Receipt
**What to do:** Compare user's stated expense (from conversation context) with extracted receipt data
**Flag:** Category mismatches (user said "hotel" but receipt is restaurant), amount discrepancies, date inconsistencies
**Behavior:** Receipt data takes precedence. If mismatch, ask user to clarify but proceed with receipt data if user confirms.
**Output format:** "Step 3: Cross-referencing your description with receipt data... [result: ✓ match / ⚠ mismatch]"
**Example (mismatch):**
```
Step 3: Cross-referencing your description with receipt data...

⚠ Inconsistency detected:
- You described: "hotel accommodation for 1 night"
- Receipt shows: DIG Restaurant (food items)

The receipt will take precedence. Do you want to:
1. Confirm this is a restaurant expense (not hotel)
2. Upload a different receipt

Please clarify.
```

### Step 4: Prepare Claim
**What to do:** Fill all mandatory submission attributes from extracted data
**Output format:** "Step 4: Preparing claim from extracted data..."

### Step 5: Currency Conversion
**What to do:** If currency != SGD, convert and show both amounts with exchange rate
**Output format:** "Step 5: Converting currency... [result: original amount + converted SGD]"
**Example:**
```
Step 5: Converting currency...

Original: USD 75.00
Converted: SGD 101.25
Exchange rate: 1.35
Date: 2026-03-24

Conversion complete.
```

### Step 6: Identify Gaps + Low Confidence Fields
**What to do:** Flag missing mandatory data AND uncertain extractions (confidence < 0.7) together
**Output format:** List all gaps and low-confidence fields in one message

### Step 7: Clarifications
**What to do:** Ask user to confirm/correct ALL flagged items in one pass (not one at a time)
**Output format:** Numbered list of questions, wait for single response

### Step 8: Check Policy
**What to do:** Search policies via RAG for relevant clauses
**Output format:** Table of retrieved policy clauses with section references
**Example:**
```
Step 8: Checking expense policies...

| Policy Clause                              | Section | Relevance |
|-------------------------------------------|---------|-----------|
| Meals: SGD 100/day limit per person       | 3.1     | 0.95      |
| Receipts required for claims > SGD 50     | 2.4     | 0.88      |

Policy search complete.
```

### Step 9: Flag Violations + Ask for Justification
**What to do:** Evaluate claim against retrieved policies. IMPORTANT: Compare numbers correctly (98.56 < 100 means NO violation).
**Output format:** List violations with cited clauses. If violations exist, ask user for justification.
**Example (correct evaluation):**
```
Step 9: Evaluating claim against policies...

Claim amount: SGD 98.56
Policy limit: SGD 100/day (Section 3.1)

✓ No violations detected (98.56 < 100).
```

### Step 10: Confirm
**What to do:** Show finalized claim summary card with all mandatory fields, ask for explicit confirmation
**Output format:** Structured table with claim submission attributes (not all extracted fields)

### Step 11: Submit
**What to do:** Persist claim + receipt to database
**Output format:** "Step 11: Submitting claim to database..."

### Step 12: Provide Claim ID
**What to do:** Show the created claim ID to the user
**Output format:** "Step 12: Claim submitted successfully. Claim ID: [id]"

## TOOLS AVAILABLE
- extractReceiptFields(imageBytes) → dict with fields + confidence
- searchPolicies(query) → list of policy clauses
- convertCurrency(amount, fromCurrency) → converted SGD amount
- submitClaim(claimData) → claim ID
- askHuman(question, data) → pauses for user input

## CONSTRAINTS
- Never skip steps, even if data seems complete
- Never combine multiple steps into one message
- Never make assumptions about missing data — always ask
- Receipt data ALWAYS takes precedence over user description in case of conflict
- If user confirms despite mismatch, accept but flag in intakeFindings
- Numeric policy evaluation must be mathematically correct (don't hallucinate violations)

Remember: You are guiding the user through a structured process. Be clear, announce each step, show all results.
"""
```

**Key insight:** 2026 prompt engineering best practice is "write success criteria + output contract + constraints." Strict numbered checklist prevents LLM from skipping steps. Few-shot examples show exact output format.

### Pattern 5: Alembic Migration for JSONB Column

**What:** Add a JSONB column to PostgreSQL table using Alembic migration with proper defaults and type imports.

**When to use:** When you need to store structured JSON data in PostgreSQL with queryability.

**Example:**
```python
# Source: Alembic documentation + PostgreSQL JSONB examples
# alembic/versions/003_add_intake_findings.py
"""Add intake_findings JSONB column to claims table

Revision ID: 003
Revises: 002
Create Date: 2026-03-25

"""
from alembic import op
import sqlalchemy as sa
from sqlalchemy.dialects import postgresql

# revision identifiers, used by Alembic.
revision = '003'
down_revision = '002'
branch_labels = None
depends_on = None


def upgrade() -> None:
    """Add intake_findings JSONB column to claims table."""
    op.add_column(
        'claims',
        sa.Column(
            'intake_findings',
            postgresql.JSONB(),
            nullable=False,
            server_default='{}'  # Empty JSON object by default
        )
    )

    # Optional: Add GIN index for faster JSONB queries
    op.create_index(
        'idx_claims_intake_findings',
        'claims',
        ['intake_findings'],
        postgresql_using='gin'
    )


def downgrade() -> None:
    """Remove intake_findings column."""
    op.drop_index('idx_claims_intake_findings', table_name='claims')
    op.drop_column('claims', 'intake_findings')
```

**Usage in submitClaim tool:**
```python
# tools/submitClaim.py
async def submitClaim(claimData: dict, intakeFindings: dict) -> dict:
    """Submit claim with intake findings to database."""

    # Include intakeFindings in DB insert
    result = await mcpCallTool(
        serverUrl=settings.db_mcp_url,
        toolName="insertClaim",
        arguments={
            "claimData": claimData,
            "intakeFindings": intakeFindings  # JSONB column
        }
    )

    return result
```

**Key insight:** PostgreSQL JSONB supports querying nested JSON fields. Use `server_default='{}'` to avoid NULL checks. GIN index speeds up JSONB queries but not required for small datasets.

### Anti-Patterns to Avoid

- **Logging after the fact:** Don't add logging only to failed code paths. Log at every major step (tool entry, MCP call, result parsing, tool exit) to trace full execution flow.

- **Single consolidated agent message:** Don't send one long message with all steps combined. User can't follow progress and doesn't know what's happening.

- **Hardcoded Seq URL:** All Seq configuration (URL, API key, ingestion port) should be in .env.local via Settings class.

- **Ignoring health check failures:** If a service fails health check during startup, script should exit immediately with error message and log location. Don't proceed with broken services.

- **Vague step announcements:** Don't say "Processing..." or "Checking policies...". Always state EXACTLY what you're doing: "Step 5: Converting USD 75.00 to SGD using Frankfurter API..."

- **System prompt assumes LLM knows best practices:** Don't rely on LLM training to handle numeric comparisons or mismatch detection. Explicitly state in prompt: "98.56 < 100 means NO violation" and "Use your judgment to flag inconsistencies between user description and receipt data."

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| JSON log formatting | Custom dict-to-JSON converter | python-json-logger JsonFormatter | Handles LogRecord attributes, timestamp formatting, extra fields, exception serialization. 4.4k GitHub stars. |
| Centralized logging UI | Custom log viewer web app | Seq Docker container (datalust/seq) | Production-ready, instant setup, powerful query DSL, free for dev use. |
| Docker health check waiting | Custom polling loop | docker compose depends_on + condition: service_healthy | Native Docker Compose feature, declarative, handles dependencies automatically. |
| Collapsible UI elements | Custom Chainlit components | cl.Step() context manager | Native Chainlit API, collapsible by default, supports nesting, typed input/output. |
| JSONB column creation | Raw SQL ALTER TABLE | Alembic migration with postgresql.JSONB() | Version-controlled schema changes, rollback support, auto-generates migrations from ORM. |
| Bash script colored output | ANSI escape codes manually | Bash color variables (e.g., RED='\033[0;31m') | Standard approach, readable, cross-platform. |
| Startup orchestration | Python subprocess calls | Bash script with set -e and functions | Universal, no dependencies, easier to read/debug for DevOps tasks. |

**Key insight:** Phase 2.2 is about wiring together existing tools, not building custom solutions. Every component (logging, health checks, Steps, migrations) has a standard library or official Docker image.

## Common Pitfalls

### Pitfall 1: Logging Doesn't Reveal submitClaim Bug

**What goes wrong:** Add JSON logging but still can't trace where dict-to-string conversion happens.

**Why it happens:** Logging added only at tool entry/exit, not inside MCP client calls or LangGraph tool node.

**How to avoid:**
- Log at EVERY major step: tool entry, before MCP call, after MCP call (with result type), before parsing result, after parsing result, tool exit
- Log the actual content: `logger.info("MCP result", extra={"result": result, "result_type": type(result).__name__})`
- Check LangGraph tool node source code — it may transform tool return values before passing to agent

**Warning signs:**
- Logs show "MCP call succeeded" but don't reveal what type was returned
- submitClaim tool logs look fine, but Chainlit runtime still crashes

### Pitfall 2: Seq Container Not Ingesting Logs

**What goes wrong:** Seq container starts healthy but shows no logs in UI.

**Why it happens:** Docker log driver not configured for JSON, or application not sending logs to Seq ingestion API.

**How to avoid:**
- Option 1 (simple): Let Seq auto-ingest from Docker logs. Configure Docker log driver in docker-compose.yml:
  ```yaml
  services:
    app:
      logging:
        driver: "json-file"  # Default, Seq can read this
  ```
- Option 2 (advanced): Send logs to Seq HTTP ingestion API (port 5342) using Serilog or custom handler
- Verify Seq is receiving logs: Check Seq UI → Events → Stream

**Warning signs:**
- Seq UI shows 0 events despite services running
- Console logs are JSON but Seq is empty

### Pitfall 3: Chainlit Steps Not Collapsing

**What goes wrong:** Steps render in UI but don't collapse/expand, just show as regular messages.

**Why it happens:** Chainlit version too old, or `config.ui.cot` setting misconfigured in chainlit.toml.

**How to avoid:**
- Ensure Chainlit >= 2.0 (Steps API added in 2.0)
- Check chainlit.toml: `[UI]` section should have `cot = "tool_calls"` or `cot = "full"`
- Never set `cot = "hidden"` if you want Steps visible (hidden mode disables all Steps)

**Warning signs:**
- Steps show as flat messages with no collapse icon
- Nested Steps don't render hierarchically

### Pitfall 4: Startup Script Succeeds but Services Not Ready

**What goes wrong:** Script exits successfully but Alembic migrations fail or RAG ingestion times out.

**Why it happens:** Health checks pass but service isn't fully ready (e.g., Postgres accepts connections but database not initialized).

**How to avoid:**
- Use `service_healthy` condition in docker-compose.yml (already present)
- For Alembic migrations, add retry logic: `docker compose exec -T app bash -c "for i in {1..5}; do alembic upgrade head && break || sleep 2; done"`
- For RAG ingestion, check Qdrant collection exists before ingesting: `curl http://localhost:6333/collections/expense_policies`

**Warning signs:**
- Script says "✓ System ready" but Chainlit app crashes on startup
- Migrations fail with "database does not exist" despite Postgres healthy

### Pitfall 5: System Prompt Too Long, LLM Ignores Instructions

**What goes wrong:** System prompt with 12-step checklist is 3000+ tokens, LLM skips steps or deviates from format.

**Why it happens:** Long prompts dilute attention, LLM focuses on first/last sections.

**How to avoid:**
- Keep system prompt under 1500 tokens (measure with tiktoken)
- Use CAPS for critical instructions: "IMPORTANT: 98.56 < 100 means NO violation"
- Put success criteria and output contract at the TOP (before step definitions)
- Use few-shot examples only for complex formats (don't example every step)

**Warning signs:**
- Agent skips step 3 (cross-reference) consistently
- Numeric policy evaluation is still wrong despite explicit instruction

### Pitfall 6: intakeFindings Dict Structure Not Defined

**What goes wrong:** submitClaim passes `intakeFindings={}` but doesn't know what keys to include.

**Why it happens:** No schema defined for intakeFindings, left as free-form dict.

**How to avoid:**
- Define Pydantic model or TypedDict for intakeFindings:
  ```python
  class IntakeFindings(TypedDict):
      mismatches: list[dict]  # {type: "category", user_said: "hotel", receipt_shows: "restaurant"}
      overrides: list[dict]   # {field: "merchant", user_confirmed: "DIG Restaurant", confidence: 0.92}
      red_flags: list[str]    # ["duplicate receipt detected", "amount exceeds threshold"]
      low_confidence: list[str]  # ["date", "payment_method"]
      user_justifications: dict[str, str]  # {"policy_3.1": "client dinner, 5 attendees"}
  ```
- Store in ClaimState as typed field: `intakeFindings: IntakeFindings`
- Pass to submitClaim tool and include in DB JSONB column

**Warning signs:**
- submitClaim crashes with KeyError when trying to access intakeFindings["mismatches"]
- Downstream agents (Compliance, Fraud) can't parse intakeFindings

### Pitfall 7: Alembic Migration Runs Before Postgres Ready

**What goes wrong:** Startup script runs `alembic upgrade head` immediately after `docker compose up`, fails with connection refused.

**Why it happens:** Script doesn't wait for health checks before running migrations.

**How to avoid:**
- In startup script, always wait for health checks BEFORE migrations:
  ```bash
  # Wait for all services
  waitForHealthy postgres
  waitForHealthy app

  # THEN run migrations
  docker compose exec -T app alembic upgrade head
  ```
- Add retry logic to migrations in case of transient failures

**Warning signs:**
- Alembic error: "connection refused" or "database does not exist"
- Script exits with error on first run, succeeds on second run

### Pitfall 8: Agent Sends Messages Too Fast, User Can't Follow

**What goes wrong:** Agent sends 12 separate messages in rapid succession (< 1 second), UI floods with steps.

**Why it happens:** No delay between steps, all processing happens instantly.

**How to avoid:**
- Option 1: Let Steps handle it (Chainlit buffers messages naturally)
- Option 2: Add small delay between user-facing messages (NOT between tool calls): `await asyncio.sleep(0.5)`
- Option 3: Use streaming — send step results incrementally as they complete

**Warning signs:**
- User reports "messages flying by too fast to read"
- Step N+1 renders before user can expand Step N

### Bash Health Check Wait Function

```bash
# Source: Docker Compose health check examples
waitForHealthy() {
    local service=$1
    local timeout=${2:-120}
    local interval=${3:-5}
    local elapsed=0

    echo "Waiting for $service to be healthy..."

    while [ $elapsed -lt $timeout ]; do
        # Get health status from Docker inspect
        status=$(docker inspect --format='{{.State.Health.Status}}' $(docker compose ps -q $service) 2>/dev/null || echo "starting")

        case $status in
            healthy)
                echo "✓ $service is healthy"
                return 0
                ;;
            unhealthy)
                echo "✗ $service is unhealthy"
                docker compose logs --tail=20 $service
                return 1
                ;;
            starting)
                echo -ne "\r⏳ $service: starting... (${elapsed}s)"
                ;;
            *)
                echo -ne "\r⏳ $service: $status (${elapsed}s)"
                ;;
        esac

        sleep $interval
        elapsed=$((elapsed + interval))
    done

    echo "\n✗ $service health check timeout (${timeout}s)"
    return 1
}

# Usage
waitForHealthy postgres 120 5 || exit 1
waitForHealthy qdrant 120 5 || exit 1
```
