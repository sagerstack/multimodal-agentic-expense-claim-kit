## Summary

Researched how leading chat products (ChatGPT, Claude, Perplexity, Gemini) implement two-layer conversational experiences—conversational output to the user + expandable thinking panel showing chain-of-thought reasoning—and how to implement this pattern with Chainlit 2.x + LangGraph for the expense claims intake agent.

The research confirms this is an established UX pattern in production AI chat products as of 2026, with consistent design principles across platforms:

1. **Layer 1 (Primary)**: Conversational, professional assistant tone—no numbered steps, tables for data, natural language flow
2. **Layer 2 (Secondary)**: Collapsible "Thinking..." panel with elapsed time indicator (ChatGPT-style), expands to show full tool calls and LLM reasoning

Chainlit supports this pattern through `cl.Step` (collapsible reasoning) + `config.ui.cot` settings + message filtering. System prompts can separate internal reasoning from user-facing output through role framing and output instructions.

**Primary recommendation:** Use Chainlit's Step context manager with `type="tool"` for thinking panel, capture filtered-out messages (ToolMessages, intermediate AI messages) as Step output, system prompt instructs conversational user output with internal 12-step reasoning guidance, and track message count before graph invoke to send only new messages (no duplication).

## Architecture Patterns

### Recommended Message Flow Structure

Current project structure:
```python
# app.py message handling
@cl.on_message
async def on_message(message: cl.Message):
    # 1. Get graph, state
    # 2. Process image if attached
    # 3. Build HumanMessage
    # 4. Invoke graph in Step context
    # 5. Check for interrupt
    # 6. Send ALL messages from result["messages"] ← DUPLICATION BUG
```

### Pattern 1: Track Message Count for Filtering

**What:** Before graph invoke, get current message count from state. After invoke, slice `result["messages"]` to only new messages.

**When to use:** Every LangGraph + Chainlit integration where the graph maintains stateful conversation history via checkpointing.

**Example:**
```python
# Source: Research synthesis, standard LangGraph checkpointing pattern
@cl.on_message
async def on_message(message: cl.Message):
    graph = cl.user_session.get("graph")
    threadId = cl.user_session.get("thread_id")

    # Get current message count from graph state
    try:
        currentState = await graph.aget_state(config={"configurable": {"thread_id": threadId}})
        messageCountBefore = len(currentState.values.get("messages", []))
    except Exception:
        messageCountBefore = 0  # First invoke, no prior messages

    # Invoke graph
    result = await graph.ainvoke(inputState, config={"configurable": {"thread_id": threadId}})

    # Slice to only new messages from this turn
    newMessages = result["messages"][messageCountBefore:]

    # Process only newMessages
    for msg in newMessages:
        if msg.type == "ai" and msg.content:
            await cl.Message(content=msg.content).send()
```

**Why this works:** LangGraph with checkpointing appends new messages to the conversation history. `result["messages"]` contains the FULL history (all turns). Slicing by messageCountBefore extracts only messages added by the current invoke.

### Pattern 2: Separate CoT Capture from User-Facing Messages

**What:** Within `cl.Step` context, separate messages into:
- **CoT entries**: ToolMessages, AI messages with only tool_calls (no content), intermediate reasoning
- **User-facing entries**: AI messages with content intended for user

Write CoT entries to `step.output`. Send user-facing messages AFTER step context exits.

**When to use:** Every agent that uses tools and you want to show tool calls/results in expandable panel.

**Example:**
```python
# Source: Research synthesis combining Chainlit Step docs + UX patterns
async with cl.Step(name="Processing Receipt", type="tool") as step:
    step.input = "Extracting receipt fields, validating policy, checking for conflicts"

    result = await graph.ainvoke(...)

    newMessages = result["messages"][messageCountBefore:]

    cotEntries = []
    userFacingMessages = []

    for msg in newMessages:
        if msg.type == "tool":
            # ToolMessage: capture result preview for CoT
            toolName = getattr(msg, "name", "unknown")
            toolContent = str(msg.content)[:500]  # Truncate for readability
            cotEntries.append(f"**Tool Result ({toolName}):** {toolContent}")

        elif msg.type == "ai":
            if hasattr(msg, "tool_calls") and msg.tool_calls:
                # AI message with tool calls
                for tc in msg.tool_calls:
                    toolName = tc.get("name", "unknown")
                    toolArgs = json.dumps(tc.get("args", {}))[:200]
                    cotEntries.append(f"**Tool Call:** {toolName}({toolArgs})")

            if msg.content:
                # AI message with user-facing content
                userFacingMessages.append(msg)

    # Write CoT to step output (visible in collapsed panel)
    step.output = "\n\n".join(cotEntries) if cotEntries else "No intermediate reasoning captured"

# OUTSIDE step context: send user-facing messages to chat
for msg in userFacingMessages:
    await cl.Message(content=msg.content).send()
```

**Why separation matters:**
- Step output displays in collapsible panel (Layer 2 — CoT)
- cl.Message appears in main chat (Layer 1 — conversational flow)
- User sees answers first, can expand Step to see tool calls/reasoning on demand

### Pattern 3: System Prompt Two-Layer Structure

**What:** System prompt explicitly separates:
1. **USER-FACING OUTPUT section**: Instructions for conversational tone, natural language, no numbered steps
2. **INTERNAL REASONING section**: 12-step guidance for agent's decision-making process

**When to use:** Any conversational agent that needs to follow internal steps but present natural output.

**Example:**
```python
# Source: Synthesis of prompt engineering best practices (2026)
SYSTEM_PROMPT = """
ROLE:
You are an expense claims assistant for SUTD. You help claimants submit expense claims by processing receipt images. You communicate in a professional, conversational tone—like a helpful colleague, not a system executing a checklist.

USER-FACING OUTPUT:
Your messages to the user are CONVERSATIONAL. Never show "Step N:" or numbered steps. Speak naturally.

Typical message flow for a clean receipt (no issues):
1. Extraction result: "Here's what I extracted from your receipt:" + markdown table
2. Pre-submission summary: Full claim summary + policy check + "Ready to submit?"
3. Confirmation: "Claim [ID] submitted successfully! Your manager will review within 48 hours."

Additional messages for issues:
- Low-confidence fields: "I couldn't read the [field] clearly. What is the [field]?"
- Policy violations: "This exceeds the daily meal cap (SGD 100). Would you like to proceed anyway? If so, please provide a brief justification."
- Description conflicts: "I noticed a discrepancy... Which would you like to use? 1. Receipt data (recommended) 2. Re-upload 3. Override"

Be autonomous: process everything without pausing unless there's a REASON to pause (low-confidence, violations, conflicts).
Before submission: ALWAYS pause for explicit user confirmation.

INTERNAL REASONING:
Follow these 12 steps internally as your reasoning process. These steps guide your tool use and decision-making. They are NEVER shown to the user.

1. Receipt quality check (validate resolution, blur)
2. Extract fields via extractReceiptFields
3. Cross-reference description vs receipt — CONDITIONAL: Only if user provided expense description in conversation. If user only uploaded image with no descriptive text, SKIP this step entirely.
4. Prepare claim attributes (fill all mandatory fields)
5. Currency conversion (if non-SGD, convert and show both amounts)
6. Identify gaps + low confidence fields (flag missing data, uncertain extractions)
7. Clarifications (ask user to confirm/correct flagged items in one pass)
8. Check policy via searchPolicies (find relevant clauses)
9. Flag violations — CRITICAL: Always compare numbers correctly. Example: claim SGD 98.56, limit SGD 100 → 98.56 < 100 = NO violation. Show math in reasoning, present conversationally to user.
10. Confirm (show summary card, ask for explicit confirmation)
11. Submit via submitClaim
12. Provide claim ID + next steps

TOOLS:
[tool signatures...]

CONSTRAINTS:
- Never show "Step N:" to user — steps are internal reasoning only
- Never combine extraction result and summary into one message — send separately
- Receipt data takes precedence over user description in conflicts
- Accumulate intakeFindings throughout: mismatches, overrides, low-confidence flags, violations, justifications
- Always show explicit numeric comparison in reasoning (98.56 < 100 = NO violation)
- If user confirms despite mismatch or violation, accept but flag in intakeFindings
- Show confidence as High/Medium/Low to user (not raw numbers like 0.95)
"""
```

**Why this works:**
- LLMs respond to explicit structure and role framing
- "INTERNAL REASONING" section provides step-by-step guidance without mandating numbered output
- "USER-FACING OUTPUT" section overrides default step-enumeration behavior
- Few-shot examples (not shown above) demonstrate conversational format

### Pattern 4: Chainlit Step Configuration

**What:** `cl.Step(name="...", type="tool")` creates collapsible thinking panel. Set `config.ui.cot = "full"` in Chainlit config to display steps by default (user can collapse). Set `type="tool"` for tool-related reasoning.

**When to use:** Wrapping any LangGraph agent invoke that includes tool calls.

**Example:**
```python
# Source: Chainlit official docs + Context7 query results
# In config.toml:
[UI]
cot = "full"  # Options: "full" (show all), "tool_call" (only tool steps), "hidden" (hide all)

# In app.py:
async with cl.Step(name="Processing Receipt", type="tool") as step:
    step.input = "Extracting fields, validating policy, converting currency"

    result = await graph.ainvoke(...)

    # Capture CoT as step.output
    step.output = "[ToolMessage summaries go here]"
```

**Available type options:**
- `"tool"` — for tool executions (icon: wrench)
- `"llm"` — for LLM API calls (icon: sparkles)
- `"run"` — for agent runs / general execution (icon: play)
- `"undefined"` — default (generic icon)

**Step lifecycle:**
- Displays "Thinking..." or step name during execution (user sees spinner)
- On exit, collapses automatically (user can expand)
- `step.input` / `step.output` populate collapsible content
- Elapsed time tracked automatically by Chainlit (start when context entered, end when exited)

### Anti-Patterns to Avoid

**1. Numbered Steps in User Output**
```python
# BAD: System prompt mandates
"Step 1: Extract receipt fields
Step 2: Check policy
..."
```
**Why it's bad:** Feels like watching a checklist execute, not talking to a professional assistant. User doesn't care about internal process.

**Fix:** Frame steps as internal reasoning guidance, instruct conversational output.

**2. Sending Full Conversation History on Every Turn**
```python
# BAD:
for msg in result["messages"]:  # ALL messages from ALL turns
    await cl.Message(content=msg.content).send()
```
**Why it's bad:** Re-displays all prior messages on each user interaction. Chat fills with duplicates.

**Fix:** Track message count before invoke, slice to new messages only.

**3. Discarding Tool Calls and CoT**
```python
# BAD:
processingStep.output = "Processing complete"  # Static string
```
**Why it's bad:** Transparency lost. User can't see what the agent actually did (tool calls, reasoning steps).

**Fix:** Capture filtered-out messages (ToolMessages, tool calls) and write to step.output.

**4. Auto-filling Low-Confidence Fields**
```python
# BAD: System prompt says
"If confidence < 0.8, use best guess but mark as uncertain"
```
**Why it's bad:** Clean data is more important than convenience. Bad extraction propagates to database.

**Fix:** Leave low-confidence fields blank, ask user directly. No threshold-based guessing.

**5. Blocking Submission on Policy Violations**
```python
# BAD:
"You cannot submit this claim. It violates policy [X]."
```
**Why it's bad:** Rigid gate prevents legitimate overrides. User escalates to manager manually (slow).

**Fix:** Warn + allow override. Capture justification in intakeFindings for reviewer.

**6. Asking Clarifying Questions for Routine Tasks**
```python
# BAD: After extraction with high confidence:
"I extracted these fields. Does this look correct? Please confirm."
```
**Why it's bad:** Unnecessary pause. Adds cognitive load for routine tasks. 2026 trend: "users don't want to converse, they want to be done."

**Fix:** Autonomous by default. Only pause if there's a reason (low confidence, conflicts, violations, confirmation before submission).

**7. Over-reliance on Conversation for Structured Data**
```python
# BAD: Text-only output for structured claim
"Your claim has merchant Starbucks, date 2024-03-15, amount SGD 12.50..."
```
**Why it's bad:** Slow to scan, hard to verify.

**Fix:** Use markdown tables for structured data. Conversational framing + table content.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Message deduplication | Custom message ID tracking, manual diffing | LangGraph state message count slicing | LangGraph checkpointing already maintains full history; slicing by count is native operation |
| CoT capture | Custom logging, separate DB table | Chainlit Step output | Step API designed for this; collapsible UI built-in; no persistence needed for CoT (ephemeral) |
| Elapsed time tracking | Manual timers, datetime.now() diffs | Chainlit Step lifecycle | Step automatically tracks start/end; UI displays elapsed time on collapse |
| Conversational tone | Hardcoded response templates | System prompt role framing + few-shot examples | LLMs excel at natural language; templates brittle, don't adapt to context |
| Multi-model orchestration (if needed later) | Custom routing logic | Perplexity-style model council | 2026 best practice for specialized AI tasks; no single model wins at everything |

**Key insight:** Chainlit + LangGraph together provide message history management, CoT visibility, and streaming out of the box. Custom solutions for these are reinventing the wheel and miss UX affordances (collapsible panels, elapsed time, syntax highlighting).

## Common Pitfalls

### Pitfall 1: Explicit CoT Prompting with Reasoning Models

**What goes wrong:** Adding "think step by step" to prompts when using reasoning models (Claude Extended Thinking, GPT o-series, Gemini Thinking Mode) degrades performance.

**Why it happens:** Reasoning models already perform internal CoT. Explicit prompting creates a conflict—model does CoT twice or prioritizes explicit instruction over internal mechanism.

**How to avoid:** Check if model has native reasoning mode. If yes, skip explicit CoT prompts. If using standard LLM (GPT-4, Sonnet, etc.), explicit CoT boosts performance (~19 points on MMLU-Pro).

**Warning signs:** Performance drop when switching from standard to reasoning model with same prompt.

**For this project:** OpenRouter models are standard (no native reasoning mode), so explicit internal reasoning guidance in system prompt is appropriate.

### Pitfall 2: Hidden CoT Prompt Conflicts

**What goes wrong:** Instruction-tuned models have hidden CoT prompts. User adds "Please only provide the final answer" → conflict → degraded output.

**Why it happens:** Model training includes implicit "respond step by step" instruction. Explicit negation creates competing directives.

**How to avoid:** Positive framing over negation. "Show only the summary table" > "Don't show reasoning steps."

**Warning signs:** Model produces verbose explanations despite "concise output" instruction.

**For this project:** System prompt uses positive framing: "Your messages are conversational. Here's what to show..." rather than "Don't show Step N:".

### Pitfall 3: Mixing CoT with User-Facing Output

**What goes wrong:** Agent sends extraction table + tool calls + reasoning + summary all in one message.

**Why it happens:** No clear separation between Layer 1 (user-facing) and Layer 2 (CoT) in message handling logic.

**How to avoid:** Filter messages by type BEFORE sending. ToolMessages and AI messages with only tool_calls go to Step output. AI messages with content go to main chat.

**Warning signs:** User sees "Calling tool: extractReceiptFields" in main chat instead of in collapsed Step panel.

**For this project:** Pattern 2 (above) implements explicit separation.

### Pitfall 4: Message Duplication from Stateful Graph

**What goes wrong:** User types "yes" to confirm. All prior messages (extraction, summary) re-appear above the confirmation response.

**Why it happens:** `result["messages"]` contains full conversation history (all turns), not just new messages. Code iterates entire history on every invoke.

**How to avoid:** Track message count before invoke. Slice to only messages added by current invoke.

**Warning signs:** Chat UI grows exponentially. After 3 turns, user sees 3 copies of first message, 2 copies of second, 1 copy of third.

**For this project:** Pattern 1 (above) implements message count tracking.

### Pitfall 5: Agent Stalling on Optional Steps

**What goes wrong:** Agent completes extraction, then stops. No error, no output. User waits indefinitely.

**Why it happens:** System prompt lists Step 3 (cross-reference) as mandatory. User uploaded image without description. Agent has no description to cross-reference → cannot proceed → stalls.

**How to avoid:** Make conditional steps explicit in prompt: "CONDITIONAL: Only perform if [condition]. If [condition false], SKIP entirely."

**Warning signs:** Graph invoke returns but filter produces zero messages. Agent completed reasoning but decided not to output (waiting for input it will never get).

**For this project:** System prompt Step 3 includes: "CONDITIONAL: Only if user provided expense description. If no description, SKIP this step."

### Pitfall 6: Capability Ambiguity

**What goes wrong:** User uploads blurry receipt, types "can you fix this?". Agent responds "I cannot edit images."

**Why it happens:** User doesn't know what agent can/cannot do. System lacks visible affordances.

**How to avoid:**
- Error messages include guidance: "I couldn't read this receipt clearly. Please re-upload a clearer photo focusing on [merchant, amount, date]."
- Initial greeting lists capabilities: "I can process receipt images, validate policy, convert currency..."

**Warning signs:** User frustration, repeated failed attempts, escalation to human support.

**For this project:** Image quality failure response includes guidance + re-upload option (conversation stays open).

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|

- **Static "Step N:" output mandates in system prompts**: 2024 pattern, replaced by two-layer model (conversational + CoT)

## Open Questions

1. **Elapsed time display in Chainlit Step**
   - What we know: Step tracks start/end automatically, UI shows elapsed time on collapse
   - What's unclear: Exact format ("Thought for Xs" like ChatGPT? Or just timestamp?), customization options
   - Recommendation: Test in UI after implementation. If format doesn't match ChatGPT style, check Chainlit config or accept default.

2. **config.ui.cot behavior with nested steps**
   - What we know: `cot = "full"` shows all steps, `"tool_call"` shows only tool steps, `"hidden"` hides all
   - What's unclear: If we create sub-steps (tool calls within main processing step), does config.ui.cot cascade?
   - Recommendation: Start with single Step wrapping full graph invoke. Nested steps are complexity we don't need for v1.

3. **Streaming support with Step**
   - What we know: Chainlit supports streaming via `step.stream_token()`, LangGraph supports streaming via `.astream()`
   - What's unclear: Streaming LangGraph output within Step context while capturing CoT
   - Recommendation: Non-streaming for v1 (simpler). LangGraph invoke returns full result → parse → send. Streaming is optimization for v2 if latency becomes issue.
