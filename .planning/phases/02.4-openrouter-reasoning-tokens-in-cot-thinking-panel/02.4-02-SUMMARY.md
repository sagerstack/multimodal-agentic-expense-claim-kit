# Phase 02.4 Plan 02: DB-Generated Claim Numbers with Idempotent Submission Summary

**One-liner:** PostgreSQL sequence generates collision-free claim numbers (CLAIM-001, CLAIM-002, ...); ON CONFLICT pattern prevents duplicate submissions from same conversation

## What Was Built

Fixed BUG-009 (claim number collisions) and BUG-008 (duplicate submissions) by moving claim number generation to a database sequence and implementing idempotent upsert logic based on natural keys.

### Architecture Changes

**Before:**
- Agent generated random claim numbers (`CLAIM-{random.randint(1, 999):03d}`)
- No deduplication — multiple submitClaim calls created duplicate claims
- Collision risk: each conversation started from CLAIM-001

**After:**
- PostgreSQL sequence generates claim numbers (`nextval('claim_number_seq')`)
- Natural key idempotency: `(employeeId + merchant + receiptDate + totalAmount)` uniquely identifies a submission
- ON CONFLICT DO NOTHING returns existing claim if duplicate detected
- Agent receives claim number from DB response, never generates it

### Database Schema

**New sequence:**
```sql
CREATE SEQUENCE claim_number_seq START WITH 1
```

**claim_number DEFAULT:**
```sql
ALTER COLUMN claim_number SET DEFAULT 'CLAIM-' || LPAD(nextval('claim_number_seq')::TEXT, 3, '0')
```

**New column:**
- `idempotency_key` (String(200), unique index)

**Backfill strategy:**
- Existing claims: `idempotency_key = id::text || '_legacy'`
- New claims: `idempotency_key = "{employeeId}_{merchant}_{receiptDate}_{totalAmount}"`

### MCP Server Changes

**insertClaim signature:**
- `claimNumber` → optional (was required)
- `idempotencyKey` → new parameter

**Idempotent insert logic:**
```python
if idempotencyKey:
    # INSERT ... ON CONFLICT (idempotency_key) DO NOTHING RETURNING *
    # If 0 rows returned → duplicate, SELECT existing claim
    # Return {"claim": {...}, "note": "Duplicate submission detected"}
else:
    # Standard INSERT (backwards compatibility)
```

**DB-generated claim numbers:**
- If `claimNumber=None`, DB DEFAULT (sequence) generates it
- Agent never provides claimNumber (legacy path logs warning)

### Agent Changes

**submitClaim tool:**
- Removed `import random`
- Compute `idempotencyKey` from natural key fields
- Removed random claim number generation (`CLAIM-{random.randint(...)}`)
- Removed random receipt number generation → deterministic `REC-{date}` or `REC-{hash}`

**System prompt:**
- Phase 2: Removed "Generate a claim number in format CLAIM-NNN"
- Phase 2: Removed CLAIM-042 from example summary table
- Phase 3: Updated to "receives claim number from submitClaim response"
- Phase 3: Added constraint "Never generate or guess a claim number"

## Technical Decisions

### 1. Natural Key for Idempotency

**Decision:** Use `(employeeId, merchant, receiptDate, totalAmount)` as natural key

**Rationale:**
- Same employee submitting same receipt twice within conversation = duplicate
- More robust than threadId (conversations can be retried)
- Deterministic without requiring agent to track state

**Alternative considered:** threadId-based key
- Rejected: multiple employees could submit same receipt in different threads

### 2. Sequence Start Value

**Decision:** Start sequence at 1 (produces CLAIM-001)

**Rationale:**
- Simple sequential numbering from project start
- Existing test claims numbered manually (CLAIM-001, CLAIM-002) preserved via backfill

**Migration safety:**
- Backfill assigns `idempotency_key = id::text || '_legacy'` to existing claims
- New sequence-generated claims start from next available number
- No collision with existing test data

### 3. Deterministic Receipt Numbers

**Decision:** Generate receipt numbers from date (`REC-20260328`) or hash fallback

**Rationale:**
- Reproducibility for testing
- No collision risk (receipt linked to claim_id FK)
- Removes random.randint() dependency

**Alternative considered:** Keep random receipt numbers
- Rejected: inconsistent with deterministic claim number approach

## Bugs Fixed

### BUG-009: CLAIM-NNN collision across sessions
**Status:** RESOLVED

**Before:**
- Agent generated sequential claim numbers starting from CLAIM-001 each session
- Each new conversation created collision with previous session's claims
- Error: `duplicate key value violates unique constraint "claims_claim_number_key"`

**After:**
- Database sequence guarantees unique incremental claim numbers
- No collision possible — sequence state persists across sessions

### BUG-008: submitClaim called twice on unique constraint violation
**Status:** RESOLVED

**Before:**
- Model attempted CLAIM-001 → collision → self-recovered with CLAIM-002
- Two DB round-trips, one failure, duplicate submission within conversation

**After:**
- ON CONFLICT pattern detects duplicate on idempotency key
- Returns existing claim immediately with note field
- Agent receives same claim data on retry (same natural key)

## Lessons Learned

1. **Sequences for unique IDs:** Auto-incrementing sequences are the correct pattern for unique identifiers; random generation leads to collisions at scale

2. **Natural keys for idempotency:** Business-meaningful keys (employee + merchant + date + amount) more robust than technical keys (threadId, sessionId)

3. **ON CONFLICT pattern:** PostgreSQL's ON CONFLICT DO NOTHING RETURNING idiom is the standard upsert pattern for idempotent writes

4. **Deterministic test data:** Removing `random.randint()` improves test reproducibility and debugging
