# Phase 2.4: CoT Thinking Panel + Bug Fixes - Research

**Researched:** 2026-03-28
**Domain:** OpenRouter reasoning tokens, Chainlit theming, PostgreSQL sequences, SSL certificates, LangGraph streaming
**Confidence:** HIGH (OpenRouter/PostgreSQL), MEDIUM (LangGraph detection), HIGH (Chainlit/SSL)

## Summary

Phase 2.4 fixes 5 open bugs from Phase 2.3 UAT and adds reasoning token support to the CoT thinking panel. The research covers six technical domains:

1. **OpenRouter reasoning tokens**: OpenRouter provides a `reasoning` parameter that enables extended thinking for supported models. However, the current codebase uses `ChatOpenAI` with base_url override, which doesn't preserve non-standard response fields like `reasoning_details`. Migration to `langchain-openrouter` package is recommended but not required — reasoning tokens can be added incrementally when/if that migration happens later.

2. **Chainlit theme-aware CSS**: Chainlit uses HSL-formatted CSS variables with separate `light` and `dark` theme definitions. The current CSS uses `rgba(255,255,255,...)` which is invisible in light mode. Fix: use `@media (prefers-color-scheme: light/dark)` with appropriate colors for each mode.

3. **PostgreSQL sequences for claim numbers**: Standard pattern is `CREATE SEQUENCE` + `DEFAULT 'CLAIM-' || LPAD(nextval('seq')::TEXT, 3, '0')` on the claim_id column. DB generates the number; agent sees it in the INSERT RETURNING response.

4. **PostgreSQL idempotent upserts**: Use `INSERT ... ON CONFLICT (idempotency_key) DO NOTHING RETURNING *` to atomically check-and-claim. If RETURNING yields no rows, the key already existed — fetch with `SELECT ... FOR UPDATE SKIP LOCKED` for non-blocking behavior.

5. **Docker SSL certificates**: Add `ca-certificates` package to Dockerfile and ensure `certifi` Python package is up to date. Set `REQUESTS_CA_BUNDLE=/etc/ssl/certs/ca-certificates.crt` in the container environment.

6. **LangGraph finalResponse detection**: The `on_chat_model_end` event includes `output.tool_calls` attribute. If present and non-empty, it's intermediate reasoning; if absent/empty, it's the final response. Current code correctly implements this pattern (lines 185-204 in app.py).

**Primary recommendation:** Fix the five bugs using standard patterns. For reasoning tokens, the infrastructure is ready (astream_events v2 already captures all events), but actual reasoning token display should be deferred until the team decides whether to migrate from ChatOpenAI to ChatOpenRouter. The phase can succeed without reasoning tokens by focusing on the five critical bugs.

## Architecture Patterns

### Pattern 1: DB-Generated Claim Numbers

**What:** PostgreSQL sequence generates claim numbers on INSERT; agent receives number in response
**When to use:** Any user-facing ID that needs to be sequential and collision-free

**Current approach (agent generates):**
```python
# Agent prompt generates CLAIM-NNN → collision risk
claimNumber = "CLAIM-001"  # Model generates this
```

**Recommended approach (DB generates):**
```sql
-- Migration: Create sequence
CREATE SEQUENCE claim_number_seq START WITH 1;

-- Alter table to use sequence
ALTER TABLE claims
    ALTER COLUMN claim_number
    SET DEFAULT 'CLAIM-' || LPAD(nextval('claim_number_seq')::TEXT, 3, '0');

-- Usage in MCP server
INSERT INTO claims (..., claim_number)
VALUES (..., DEFAULT)
RETURNING claim_number;  -- Returns CLAIM-001, CLAIM-002, etc.
```

**Source:** [PostgreSQL: Implementing custom human-readable IDs](https://medium.com/@kibzrael/postgresql-implementing-custom-human-readable-ids-b946c75c917a)

### Pattern 2: Idempotent Upsert with ON CONFLICT

**What:** Atomic check-and-claim pattern prevents duplicate submissions
**When to use:** Any operation that must be idempotent (payment processing, claim submission)

**Implementation:**
```python
# Generate idempotency key from natural identifiers
idempotencyKey = f"{claimId}_{threadId}"

# Atomic insert-or-fetch
result = await conn.execute(
    """
    INSERT INTO claims (claim_id, thread_id, idempotency_key, ...)
    VALUES ($1, $2, $3, ...)
    ON CONFLICT (idempotency_key) DO NOTHING
    RETURNING *
    """,
    claimId, threadId, idempotencyKey, ...
)

if result.rowcount == 0:
    # Key already existed, fetch existing claim
    existing = await conn.fetchone(
        "SELECT * FROM claims WHERE idempotency_key = $1 FOR UPDATE SKIP LOCKED",
        idempotencyKey
    )
    return existing
else:
    return result.fetchone()
```

**Source:** [Implementing Stripe-like Idempotency Keys in Postgres](https://brandur.org/idempotency-keys)

### Pattern 5: LangGraph Final Response Detection

**What:** Distinguish final response from intermediate reasoning in `on_chat_model_end` events
**When to use:** Any ReAct agent that streams responses and needs to display only the final answer

**Current implementation (correct):**
```python
elif eventKind == "on_chat_model_end":
    output = event.get("data", {}).get("output")
    hasToolCalls = (
        output
        and hasattr(output, "tool_calls")
        and output.tool_calls
    )

    if hasToolCalls:
        # Intermediate thinking — discard narration tokens
        tokenBuffer = ""
    else:
        # Final response — capture for replay
        finalResponse = tokenBuffer
        tokenBuffer = ""
```

**Key insight:** The `output.tool_calls` attribute is the definitive signal. Present = intermediate, absent = final.

**Source:** [LangGraph ReAct Agent: Tool-Calling from Scratch](https://markaicode.com/langgraph-react-agent-tool-calling/)

### Anti-Patterns to Avoid

- **Agent-generated claim numbers:** Race conditions and collisions inevitable with LLM-generated sequential IDs
- **`rgba(255,255,255,...)` for panels:** Invisible in light mode; always use theme-aware colors
- **Manual SSL certificate installation:** Use package managers (`apt install ca-certificates`) not curl/wget downloads
- **Checking `stop_reason` during streaming:** Not available until message is complete; use `tool_calls` attribute instead

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Sequential IDs with prefix | Python counter class | PostgreSQL SEQUENCE + DEFAULT | Sequences are atomic, survives restarts, handles concurrency |
| Idempotency keys | In-memory dedup dict | PostgreSQL ON CONFLICT | Need persistence, atomic check-and-claim, works across instances |
| Theme detection | JavaScript theme toggle | CSS @media (prefers-color-scheme) | Automatic, no JS needed, respects system preference |
| CA certificates | Custom download script | apt install ca-certificates | Package manager ensures updates, proper trust chain |
| Reasoning token extraction | Custom response parsing | langchain-openrouter (when ready) | Official package handles all edge cases |

**Key insight:** For Phase 2.4, focus on bug fixes using proven patterns. Reasoning token display can be added later when team is ready to migrate from ChatOpenAI to ChatOpenRouter.

## Common Pitfalls

### Pitfall 1: ChatOpenAI Doesn't Preserve reasoning_details

**What goes wrong:** You add `model_kwargs={"reasoning": {"effort": "high"}}` to ChatOpenAI, OpenRouter returns reasoning_details in the response, but LangChain silently drops them because they're not in the official OpenAI API spec.

**Why it happens:** ChatOpenAI is designed for official OpenAI API only; non-standard fields from OpenRouter/LiteLLM/etc are not extracted.

**How to avoid:** Use `langchain-openrouter` package for reasoning token support. If staying with ChatOpenAI, don't expect reasoning_details to be captured.

**Warning signs:** OpenRouter dashboard shows reasoning tokens used, but LangChain events don't contain them.

**Source:** [Reasoning tokens not passing through from OpenRouter to ChatOpenAI](https://github.com/langchain-ai/langchain/issues/32981)

### Pitfall 2: Claim Number Collision on Retry

**What goes wrong:** Agent generates CLAIM-001, INSERT fails with unique constraint violation, agent generates CLAIM-002 and retries. Database now has CLAIM-002 but user expected CLAIM-001.

**Why it happens:** LLM doesn't know what claim numbers already exist in the database; it generates sequential numbers starting from 001 every session.

**How to avoid:** Let the database generate the claim number using a sequence. Agent NEVER generates CLAIM-NNN.

**Warning signs:** Log shows "duplicate key value violates unique constraint", followed by successful INSERT with different number.

**Source:** bugs.md BUG-008, BUG-009

### Pitfall 3: CSS Colors Invisible in Light Mode

**What goes wrong:** Thinking panels work perfectly in dark mode (most developer setups), ship to production, users in light mode report "no thinking panel visible".

**Why it happens:** `rgba(255, 255, 255, 0.04)` means "white with 4% opacity" — visible on dark background, invisible on light background.

**How to avoid:** Always test custom CSS in both light and dark modes before shipping. Use `@media (prefers-color-scheme)` with inverted colors.

**Warning signs:** HTML is generated (visible in inspector), CSS classes are applied, but nothing renders visually in light mode.

**Source:** bugs.md BUG-007

### Pitfall 4: finalResponse Assignment Race Condition

**What goes wrong:** ReAct agent emits multiple `on_chat_model_end` events (one after each tool call for intermediate reasoning, one final). If you assign `finalResponse` on the first event, you capture intermediate text instead of the final answer.

**Why it happens:** ReAct loop generates text → calls tool → generates more text → repeats. Each generation triggers `on_chat_model_end`.

**How to avoid:** Only assign `finalResponse` from `on_chat_model_end` events where `output.tool_calls` is empty. Discard intermediate generations.

**Warning signs:** Agent's response starts mid-thought ("Now I'll convert the currency...") instead of addressing the user.

**Source:** bugs.md BUG-011, app.py lines 185-204

### Pitfall 5: SSL Certificate Mismatch Between Dev and Docker

**What goes wrong:** Code works fine in local dev (macOS/Linux with up-to-date certs), fails intermittently in Docker with "CERTIFICATE_VERIFY_FAILED".

**Why it happens:** Docker base images have older/missing CA certificates; host system has recent certs from OS updates.

**How to avoid:** Always install `ca-certificates` package in Dockerfile and update `certifi` Python package. Test HTTPS calls inside the container, not just on host.

**Warning signs:** Works locally, fails in Docker; works sometimes in Docker (cached connection), fails on fresh container.

**Source:** bugs.md BUG-006

Verified patterns from official sources:

### Idempotent Claim Submission

```python
# submitClaim MCP tool
async def submitClaim(claimId: str, threadId: str, totalAmount: float, ...) -> dict:
    """Submit claim with idempotent guarantee."""

    # Generate idempotency key from natural identifiers
    idempotencyKey = f"{claimId}_{threadId}"

    async with pool.connection() as conn:
        # Atomic insert-or-fetch
        result = await conn.execute(
            """
            INSERT INTO claims (
                claim_id, thread_id, idempotency_key,
                total_amount, status, submitted_at
            )
            VALUES ($1, $2, $3, $4, 'submitted', NOW())
            ON CONFLICT (idempotency_key) DO NOTHING
            RETURNING claim_number, claim_id, total_amount, status
            """,
            claimId, threadId, idempotencyKey, totalAmount
        )

        if result.rowcount > 0:
            # New claim inserted
            claim = result.fetchone()
            logger.info(f"Claim submitted: {claim['claim_number']}")
            return {
                "success": True,
                "claimNumber": claim["claim_number"],
                "status": claim["status"]
            }
        else:
            # Idempotency key already existed, fetch existing
            existing = await conn.fetchone(
                """
                SELECT claim_number, claim_id, total_amount, status
                FROM claims
                WHERE idempotency_key = $1
                FOR UPDATE SKIP LOCKED
                """,
                idempotencyKey
            )
            logger.info(f"Claim already exists: {existing['claim_number']}")
            return {
                "success": True,
                "claimNumber": existing["claim_number"],
                "status": existing["status"],
                "note": "Duplicate submission detected, returning existing claim"
            }
```

**Source:** Adapted from [Implementing Stripe-like Idempotency Keys in Postgres](https://brandur.org/idempotency-keys)

### Alembic Migration for Claim Number Sequence

```python
# migrations/versions/xxx_add_claim_number_sequence.py
"""Add claim number sequence and idempotency key

Revision ID: xxx
Revises: yyy
Create Date: 2026-03-28
"""
from alembic import op
import sqlalchemy as sa


def upgrade():
    # Create sequence
    op.execute("CREATE SEQUENCE claim_number_seq START WITH 1")

    # Add idempotency_key column
    op.add_column('claims', sa.Column('idempotency_key', sa.String(200), nullable=True))

    # Create unique index on idempotency_key
    op.create_index('idx_claims_idempotency_key', 'claims', ['idempotency_key'], unique=True)

    # Alter claim_number to use sequence as default
    op.execute("""
        ALTER TABLE claims
        ALTER COLUMN claim_number
        SET DEFAULT 'CLAIM-' || LPAD(nextval('claim_number_seq')::TEXT, 3, '0')
    """)

    # Backfill idempotency_key for existing claims (if any)
    op.execute("""
        UPDATE claims
        SET idempotency_key = claim_id || '_legacy'
        WHERE idempotency_key IS NULL
    """)

    # Make idempotency_key NOT NULL after backfill
    op.alter_column('claims', 'idempotency_key', nullable=False)


def downgrade():
    # Remove default
    op.execute("ALTER TABLE claims ALTER COLUMN claim_number DROP DEFAULT")

    # Drop index and column
    op.drop_index('idx_claims_idempotency_key', 'claims')
    op.drop_column('claims', 'idempotency_key')

    # Drop sequence
    op.execute("DROP SEQUENCE claim_number_seq")
```

**Source:** PostgreSQL official docs, Alembic patterns

## Open Questions

Things that couldn't be fully resolved:

1. **Should we migrate to langchain-openrouter now or defer?**
   - What we know: ChatOpenAI + base_url works fine for basic calls; langchain-openrouter needed for reasoning tokens
   - What's unclear: When (if ever) the team wants reasoning token display in the thinking panel
   - Recommendation: **Defer.** Fix the 5 bugs in Phase 2.4 using current stack. Schedule reasoning tokens as a separate phase when team confirms the requirement. The infrastructure (astream_events v2) is already in place; just need to decide on the migration timing.

2. **Which models on OpenRouter support reasoning tokens?**
   - What we know: DeepSeek R1, Claude 3.7+ Sonnet :thinking variant, OpenAI o1/o3/GPT-5 series, Gemini Thinking (coming soon)
   - What's unclear: Whether current models (google/gemini-2.0-flash-001, google/gemini-2.0-flash-lite-001) support reasoning tokens
   - Recommendation: Check OpenRouter model details page for `supported_parameters` containing `reasoning`. If not supported, reasoning token display gracefully falls back to tool-call-only panel (no errors).

3. **Should idempotency key include userId or just claimId + threadId?**
   - What we know: Stripe pattern uses (userId, idempotencyKey); current design has claimId tied to sessionId (1:1)
   - What's unclear: Whether claims can be shared across users or sessions in future phases
   - Recommendation: Use `{claimId}_{threadId}` for now (simplest). If multi-user access is added later, add userId to the key in that phase.
