## Tests

### 1. End-to-end claim submission
expected: Upload a receipt image in Chainlit. A "Processing" parent Step appears with spinner. Nested Thinking Steps appear per tool call. After processing, a conversational response streams in with extraction table, asks to confirm details and for employee ID. Confirming proceeds through policy check and submission. Claim + receipt records persist to database.
result: [pending]

### 2. Thinking-first UI architecture
expected: All intermediate LLM reasoning and tool calls happen inside a collapsible "Processing" parent Step. The parent Step contains nested "Thinking" child Steps per tool call. Only the final conversational response appears as a regular chat message AFTER the parent Step closes. No raw JSON or intermediate reasoning visible in chat.
result: [pending]

### 3. Continuous spinner during processing
expected: From the moment you send a message until the response appears, the spinner/busy indicator stays active. No gaps where the indicator disappears and reappears. The parent "Processing" Step keeps the spinner alive throughout.
result: [pending]

### 4. Nested Thinking Steps with tool summaries
expected: Expanding the parent "Processing" Step reveals nested child Steps. Each child shows "Thought for X seconds" as its name. Expanding a child shows what tool was called and a human-readable summary of the result (e.g., "Extracted receipt: Starbucks, SGD 12.50" not raw JSON).
result: [pending]

### 5. Token-by-token response streaming
expected: After the Thinking panel closes, the agent's response message appears with a typing effect — text streams in character by character, not appearing all at once as a block.
result: [pending]

### 6. Multi-turn conversation flow
expected: The claim process takes 3 conversation turns: (1) Upload receipt → extraction table + confirm request, (2) Confirm details → policy check + claim summary + submit request, (3) Confirm submission → claim submitted confirmation. Each turn has its own Processing Step + response.
result: [pending]

### 7. Employee ID collection
expected: Agent asks for employee ID during the first turn (extraction phase). Does NOT use placeholder text like "(your employee ID)". Waits for the actual value before proceeding to submission.
result: [pending]

### 8. Policy check with cited clauses
expected: After confirming extraction details, agent checks policies and shows a visible result: "Your [category] expense of SGD [amount] is within the [limit] (Section X.Y)" or flags a violation with the specific policy clause. This appears in the final response, not hidden in a Thinking panel.
result: [pending]

### 9. CLAIM-NNN in summary card
expected: The claim summary (shown before submission confirmation) includes a claim number in CLAIM-NNN format (e.g., CLAIM-042). This number persists after submission.
result: [pending]

### 10. No raw JSON or placeholders in chat
expected: Agent never outputs raw JSON blocks, bracket-notation placeholders like [amount] or [rate], or tool response data in the chat. All data is presented as markdown tables or conversational text.
